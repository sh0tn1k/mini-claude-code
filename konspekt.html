1. Мастер-луп (главный цикл агента)

✓ Разобрано и проверено вопросами

Исходный вопрос

«Мастер-луп — это просто мы вызываем модель по API бесконечно, пока она не вернёт выход из цикла?»

Главная мысль

Цикл останавливается по СТРУКТУРЕ ответа, а не по смыслу текста.

Обвязка не читает текст и не ищет в нём «я закончил». Она смотрит только одно: запросила ли модель инструмент (tool_use) или нет.

Каждый ответ модели бывает структурно одного из двух видов:

  • Есть tool_use (например «выполни bash», «прочитай файл») → обвязка исполняет инструмент, кладёт результат обратно в контекст, снова зовёт модель. Виток продолжается.
  • Только текст, без tool_use → модели нечего делать руками, это финальный ответ пользователю. Выход из цикла.

Это прямое следствие 1-го принципа harness engineering: обвязка никогда не ветвится на содержании ответа модели. Она «тупая» — не понимает, что задача решена, а лишь проверяет «просили инструмент? да/нет».

Полная схема одного витка

1. зовём модель (передаём весь контекст)
2. модель возвращает ход: [текст?] + [0..N tool_use?]
3. показываем текст пользователю
4. есть tool_use?
   ├─ нет  → выходим из цикла, это финальный ответ
   └─ да   → исполняем каждый инструмент,
             кладём результаты обратно в контекст,
             возвращаемся к шагу 1

Тихая страховка архитектуры

Модель физически не может ничего сделать в мире, просто написав об этом текстом. Если она пишет «Сейчас удалю файл», но не приложила tool_use на удаление — ничего не удалится, текст просто уйдёт пользователю. Любое действие = явный tool_use = проходит через обвязку и её разрешения.

Текст рядом с инструментом — «мысли вслух»

Если модель присылает текст + tool_use одновременно («Сейчас прочитаю конфиг» + read), виток продолжается (есть инструмент). А текстовая часть:

  • показывается пользователю сразу (ощущение, что агент «рассуждает на ходу»);
  • не выбрасывается — остаётся в контексте как часть хода модели. На следующем витке модель видит свой прошлый текст + результат инструмента, т.е. помнит, зачем полезла в файл. Это её рабочая память, а не испаряющаяся болтовня.

N инструментов в одном ходу → параллелизм

Модель может за один ход запросить несколько инструментов сразу. Ключ:

Эти инструменты выбраны в одном ходу, не видя результата ни одного из них → по данным они независимы по определению. Поэтому их можно параллелить (в Python — asyncio.gather).

Сама структура витка отсекает зависимости по данным: если бы шаг 2 зависел от результата шага 1, модель не смогла бы запросить их вместе — пришлось бы дождаться результата шага 1 (новый виток) и только потом запросить шаг 2.

Опасность не в зависимости по данным (её нет), а в побочных эффектах и гонках. read разных файлов — параллелим свободно. А write file.txt + read file.txt в одном ходу — гонка (старая версия или новая?). Поэтому: read-only (read, grep, glob) параллелятся свободно, а write/bash обвязка сериализует или запускает осторожнее.

Почему зависимые шаги — это минимум 2 витка

Пример: «прочитай config.json и поменяй нужную строку исходя из содержимого».

  1. Виток 1: модель присылает read config.json и отдаёт управление. Она не может в этом же ходу приложить write — ещё не знает, что в файле и что менять.
  2. Обвязка исполняет read, кладёт содержимое в контекст, снова зовёт модель.
  3. Виток 2: модель видит содержимое и присылает write с конкретным изменением.
  4. (Опционально Виток 3: повторный read, чтобы проверить результат — «агентное» поведение «сделал → проверил».)

Граница, которую нельзя перепрыгнуть: результат инструмента появляется в контексте только на следующем витке. Любая цепочка «сначала узнать → потом действовать на основе узнанного» — это минимум два витка.

Связь с управлением контекстом: большой файл читается частями (read с offset/limit) — берём ровно столько, сколько нужно, а не весь файл сразу.

Развёрнутое пояснение: зачем всё равно нужен потолок витков

Раз выход определён строго структурно (нет tool_use → стоп), кажется, что бесконечный цикл невозможен в принципе. Но модель может честно, виток за витком, находить повод позвать ещё один инструмент — не ломая контракт, а просто продолжая работу дольше, чем нужно (перечитывает файлы, перепроверяет то, что уже проверила). Это не баг структуры, а её слепое пятно: она видит «инструмент запрошен → продолжаем», но не видит «а нужно ли». Поэтому в реальных системах поверх структурного условия ставят второе, тоже структурное: жёсткий потолок числа витков (max_turns). Как только он достигнут — цикл прерывается принудительно, даже если модель прислала очередной tool_use. Это не противоречит принципу «цикл не судит смысл»: потолок тоже ничего не понимает про содержание, он просто считает витки. Защита добавляется тем же языком, которым написан весь цикл — счётчиком, а не эвристикой «похоже, застряла».

Проверка понимания (контрвопросы, на которые я ответил верно)

В: Модель пишет «Сейчас удалю файл» без tool_use — луп удалит или покажет текст?

О: Покажет текст пользователю, ничего не удалит — нет запроса на инструмент, это просто финальный текст.

В: Текст + tool_use в одном ходу — виток продолжается или выход?

О: Продолжается (есть инструмент), результат уйдёт модели обратно. Текст — как мысли вслух.

В: N инструментов в одном ходу — параллелить или по очереди?

О: Параллелить, но с оглядкой на контекст/побочные эффекты (зависимые действия или конфликт write/read).

В: «прочитай config и поменяй строку» — сколько витков минимум?

О: Минимум 2 — сначала read (возможно по кускам), затем write на основе прочитанного, возможно ещё повторный read для проверки.

2. Инструменты и диспетчеризация (реестр)

✓ Разобрано и проверено вопросами

Исходный вопрос

«Инструменты явно заданы и не привязаны жёстко — можно добавить новый, просто добавив его в список, не написав под него логику в цикле?»

Главная мысль

Цикл не знает имён инструментов. В мастер-лупе нет ни одного if name == "read_file". Связь «имя → функция» живёт в отдельном реестре (словаре), а цикл просто достаёт оттуда нужный инструмент по имени.

Каждый инструмент — это три вещи в одном объекте (Tool):

  • функция (handler) — что реально делается;
  • JSON-схема (parameters) — её отдаём модели, чтобы она знала, как вызвать;
  • флаг read_only — можно ли параллелить (read-only — да, пишущие — по очереди).

Как работает диспетчеризация

REGISTRY: dict[str, Tool] = {}        # имя → Tool

# цикл получил от модели вызов name + args:
tool = REGISTRY.get(name)             # достаём по имени
result = tool.handler(**args)         # вызываем — БЕЗ ветвлений на имя

Схемы для API собираются тоже из реестра автоматически (openai_tools_spec() проходит по REGISTRY.values()).

Добавить инструмент = одна регистрация. Пишем функцию + register(Tool(...)). И цикл, и список схем для модели подхватят новый инструмент сами. Логику в мастер-лупе трогать НЕ нужно — он остаётся «тупым» диспетчером.

Это та же идея, что и в мастер-лупе: обвязка не завязана на конкретику. Цикл не знает смысла ответа (тема 1) и не знает имён инструментов (тема 2) — он работает со структурой, а конкретика лежит в данных (реестре).

Развёрнутое пояснение: что регистрация экономит на практике

Показательно сравнить с альтернативой «в лоб» — гигантским if/elif в теле цикла, который бы разбирал имя инструмента и вручную дёргал нужную функцию. При десятке инструментов разница почти незаметна. Но при полусотне (а в реальном Claude Code — сотнях, включая MCP-инструменты из темы 21) такой блок стал бы отдельным файлом на тысячи строк, куда пришлось бы лезть при каждом изменении цикла — то есть ровно то смешение «структуры цикла» и «конкретики инструментов», которое реестр специально разводит. Реестр держит эти две вещи раздельно физически: файл цикла не меняется размером, сколько бы инструментов ни добавили, а файл с инструментами растёт независимо и не требует понимания устройства цикла, чтобы в нём разбираться.

3. Планирование (где живёт план)

✓ Разобрано и проверено вопросами

Исходный вопрос

«Про планирование — то есть сначала модель разбивает задачу на план и уже его фиксирует?»

Главная мысль

План нигде не «фиксируется». В обвязке нет переменной plan и нет механизма, который заставлял бы модель ему следовать. План — это просто текст («мысли вслух»), который модель сгенерировала сама и который осел в контексте.

Где живёт план

Нигде отдельно. Это тот самый текст хода assistant, что едет вместе со всем контекстом в messages. Для обвязки он неотличим от любого другого текста — она не знает, что это «план».

Как создаётся

Модель создаёт его сама, по своей инициативе — её подтолкнули системным промптом («сначала составь план»). Никакого вызова «создать план» нет, это просто сгенерированный первым ходом текст.

Почему модель его не бросает

На каждом витке мы отдаём модели весь накопленный контекст целиком — значит и план, и результаты уже выполненных шагов. Модель перечитывает: «шаги 1–5, шаг 1 сделан → делаю шаг 2». Её никто не заставляет — ей самой проще продолжить, потому что всё перед глазами.

Это и есть «путь наименьшего сопротивления» из статьи: модель аккуратна не по природе, а потому что структура витка (контекст копится и возвращается целиком) делает следование плану естественным. План — это «галлюцинация, ставшая полезной», потому что контекст её сохраняет.

План как инструмент — TodoWrite

Следующий шаг той же идеи: вместо того чтобы план растворялся в «мыслях вслух», его делают отдельным инструментом. Модель вызывает TodoWrite и передаёт список этапов со статусами (pending / in_progress / done). Инструмент возвращает отрисованный список ([ ] / [~] / [x]), и этот текст оседает в контексте — значит план виден модели на каждом следующем витке сам собой, бесплатно.

Почему это лучше плана в .md-файле: файл лежит на диске вне контекста — чтобы им пользоваться, нужен отдельный виток на read, и в контекст вваливается весь файл. А результат TodoWrite едет в контексте сам. Поэтому рабочий план задачи — это TodoWrite, а .md — это долговременный документ (спека, заметка), не нужный в фокусе постоянно.

Один TodoWrite «целиком» против разнесённых read / update / write

Можно было бы сделать три инструмента: todo_read (прочитать план), todo_write (записать), todo_update (точечно поменять один пункт). Но на практике (и в настоящем Claude Code) — один TodoWrite, который принимает весь список заново и перезаписывает его целиком.

  • Зачем отдельный todo_read? Не нужен: план и так в контексте после последнего TodoWrite. Отдельное чтение дублировало бы то, что уже перед глазами.
  • Чем плох точечный todo_update? Он требует адресации («пункт №7 → done»). Ошибка в номере или переупорядочивание списка → рассинхрон: обвязка думает одно, модель другое. Источник багов.
  • Почему «целиком» надёжнее. Модель шлёт «как должно быть сейчас» — состояние консистентно по построению, номером не ошибёшься.

Принцип: когда состояние маленькое — перезаписывай целиком, а не патчи точечно. План — десяток коротких строк, переслать заново стоит копейки токенов; простота и невозможность рассинхрона важнее этой экономии. Точечный патч оправдан только когда состояние большое и переслать его дорого (правка одной строки в файле на 2000 строк → edit, а не write). Та же пара «целиком vs точечно», что у write_file и edit_file.

Развёрнутое пояснение: план как побочный эффект памяти, а не декларация

Стоит явно закрыть кажущийся парадокс: если план — не более чем текст в контексте, что мешает модели просто «передумать» на середине и молча уйти в сторону? Формально — ничего, обвязка это не запрещает. Но на практике этого почти не происходит, и причина не в дисциплине модели, а в том, что TodoWrite делает план самым свежим и заметным структурированным текстом в контексте — он оседает ближе к концу истории (после каждого обновления), а не тонет где-то в середине давнего рассуждения. Модель с высокой вероятностью продолжает то, что видит перед собой яснее всего. План работает не как правило, которое нельзя нарушить, а как самый сильный сигнал в поле зрения — управление через видимость, а не через запрет.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: Если план — просто текст, кто и чем его «фиксирует»?

О: Никто. Отдельной сущности «план» нет — это текст в контексте, обвязка его смысла не понимает.

В: Если бы на каждом витке передавали только последнее сообщение, а не весь контекст — что стало бы с планированием?

О: Модель работала бы как с чистого листа: не помнила бы ни плана, ни выполненных шагов. Зацикливалась бы / повторяла шаг 1, дошла бы (если вообще) случайно за десятки витков вместо пяти. Накопленный контекст = память, превращающая отдельные вызовы в последовательное выполнение плана.

В: TodoWrite обновляет план целиком или точечно? Почему «целиком» выигрывает у точечного update?

О: Целиком — модель присылает весь список заново. Точечный update экономит токены, но требует адресации пункта → риск рассинхрона. На маленьком плане экономия копеечная, а простота и консистентность важнее. Точечный патч нужен только для большого состояния (как edit для огромного файла).

4. Субагенты и изоляция контекста

✓ Разобрано и проверено вопросами

Исходный вопрос

«Модель сама определяет, когда вызвать субагента? И это по сути просто клон основного мастер-лупа, только завершение его работы подаётся на вход в наш основной цикл? И неважно — 1 субагент или 100?»

Главная мысль

Субагент — это обычный инструмент в реестре. Для родителя он стоит рядом с read_file, grep, TodoWrite. Модель решает позвать его так же, как любой другой инструмент (tool_choice="auto" + нудж из системного промпта). Никакой особой машинерии «теперь запускаем субагента» в цикле нет.

А его handler — это рекурсивный вызов того же мастер-лупа со свежим, чистым messages[]. Внутри субагент крутит ровно те же витки, что и родитель.

Как результат возвращается родителю

Субагент крутится, пока не дойдёт до выхода из цикла (нет tool_use → финальный текст). Этот финальный текст возвращается родителю как обычное сообщение role: "tool" — садится в контекст родителя ровно на место результата того вызова, которым субагента позвали.

Что такое изоляция и зачем она

Субагент сделал внутри 40 витков (glob, десяток read_file, grep…) — в контекст родителя попадёт только финальный ответ. Все 40 промежуточных шагов отбрасываются вместе с субагентом. Родитель платит контекстом за вывод, а не за процесс.

Это и есть изоляция: шум исследования не засоряет основное окно диалога. Шаблон, который позволяет работать с произвольно большой кодовой базой — каждый промежуточный результат, не нужный для итогового ответа, остаётся внутри субагента и умирает после завершения.

Реализуется через полностью независимый messages[] у каждого субагента. Между родителем и ребёнком нет общего состояния.

Единственный «мостик» между изолированными контекстами

Если контексты независимы — как субагент узнаёт, что делать? Задачу передаёт родитель аргументом tool_call'а (run_subagent(prompt="посчитай питон-файлы…")). Внутри субагента этот аргумент кладётся в его свежий messages[] как user-сообщение (задача). Системный промпт у субагента может быть свой, общий. То есть: задача приходит аргументом вызова → становится user-сообщением субагента. Больше связи между контекстами нет.

1 субагент или 100 — и можно ли параллелить

Структурно неважно: каждый со своим messages[], общего состояния нет. Настоящий Claude Code субагентов как раз любит запускать параллельно — десяток разом, каждый смотрит свой участок базы. Но это безопасно не потому, что они субагенты:

Параллелить можно то, что только читает. Исследующий субагент внутри делает лишь glob/grep/read_file → он read-only по эффекту, ничего не пишет на диск → гонке нечего ломать. Опасность появляется ровно когда субагент начинает писать (write_file/edit_file): двух пишущих в один файл параллелить нельзя.

Критерий тот же read_only-флаг, что и для обычных инструментов (тема 1), просто поднятый на уровень целого субагента: пишет на диск или нет.

Какой флаг честно поставить субагенту

Тонкость: субагент внутри может и читать, и писать — заранее не угадаешь. Две развилки:

  • Один общий run_subagent: раз он умеет писать — честно read_only=False (всё по очереди). Безопасно, но цена — теряем параллелизм даже для чисто исследующих субагентов.
  • Два разных инструмента (так делает Claude Code): отдельный read-only «Explore»-субагент, которому внутри доступны только read-инструменты → read_only=True → параллелится десятками; и отдельный пишущий → read_only=False → по очереди.

read_only честен, только если он подтверждён набором инструментов внутри субагента, а не намерением. Read-only снаружи держится не на обещании «он не будет писать», а на том, что писать ему просто нечем.

Развёрнутое пояснение: субагент не «видит» родителя вообще

Легко представить себе субагента как процесс, у которого «где-то на фоне» доступен родительский контекст — на случай, если понадобится подсмотреть. Это неверно буквально: messages[] субагента не содержит ни одной ссылки на историю родителя, кроме того единственного текста, что пришёл аргументом вызова. Если родитель уже провёл 30 витков исследования и знает важную деталь, но не упомянул её в задаче субагенту — субагент о ней не узнает никак, даже случайно. Это не недоработка, а прямое следствие изоляции: единственный канал — аргумент вызова, значит всё нужное субагенту должно быть явно в него упаковано родителем. Отсюда практическое следствие для формулировки задачи субагенту: она должна быть самодостаточной, будто пишешь письмо человеку, который не видел переписку выше.

О: Только финальный ответ; остальные 39 отбрасываются. В этом суть изоляции — родитель платит за вывод, а не за процесс, шум не засоряет основной контекст.

Навыки и постепенное раскрытие (progressive disclosure)

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

Что такое «постепенное раскрытие»? Есть тул load_skill, модели присылается краткая сводка, какие навыки есть, и она точечно говорит «нужен этот»? То есть то же самое, что с тулами, только сам навык кладётся в контекст?

Главная мысль: навык — НЕ то же, что тул

Похоже только механикой вызова. По сути — разные звери:

  • Тул — это описание, как звать функцию. Чтобы модель могла его вызвать, обвязке нужны точные параметры → в контексте всегда лежит вся JSON-схема целиком (имя + описание + parameters со всеми полями), на каждом витке.
  • Навык — это инструкция/знание для самой модели: текст, который она прочитает и начнёт ему следовать. Исполняется не в обвязке (у него нет handler), а в голове модели.

Почему тул нельзя спрятать, а навык можно

Полную схему тула не спрячешь: модель обязана знать заранее, какие параметры передать, иначе не сформирует корректный вызов и обвязка не выполнит. А навык применяет сама модель — ей до поры не нужно знать его содержимое, только что он существует и когда его применять.

Поэтому 500 тулов с полными схемами = десятки тысяч токенов в каждом запросе, контекст забит «оглавлением» ещё до работы. А навыков может быть сколько угодно — постоянно висит лишь крошечная метастрока.

Постепенное раскрытие = уровни, а не один скачок

Слово «постепенное» — про слои детализации, которые подтягиваются по мере надобности:

  1. уровень 0 — метаданные навыка (имя + одна фраза «когда применять»), висят в контексте всегда, почти бесплатно;
  2. уровень 1 — тело навыка (SKILL.md, полная инструкция), грузится по вызову load_skill, когда модель решила «оно мне нужно»;
  3. уровень 2 — навык внутри себя может ссылаться на ещё файлы/скрипты («подробности в reference.md», «запусти build.py»), и они подтянутся, только если до них реально дойдёт дело.

Платишь токенами ровно за ту глубину, до которой дошёл, а не за всё дерево навыка сразу. Та же идея, что у read_file с offset/limit и у субагента: не тащить в контекст то, что прямо сейчас не нужно.

Как раскрытый навык попадает в контекст

Здесь тонкость — интуитивно тянет сказать «в системный промпт», но по механике это не так:

  • Канал доставки — как у тула: модель зовёт load_skill(name=...) (tool_call), и полная инструкция прилетает сообщением роли tool в messages[] — ровно тем же путём, что любой результат инструмента.
  • Содержимое — как у системного промпта: не «результат вычисления», а текст-инструкция, которой модель следует до конца задачи.

Почему нельзя зашить навык в системный промпт заранее? Системный промпт уходит первым сообщением, ещё до того как ясно, какой навык понадобится; он уже в контексте и задним числом неизменяем. Решение «мне нужен этот навык» возникает в середине диалога — а добавить что-то в идущий диалог можно только новым сообщением в messages[].

А можно ли так же лениво грузить и сами тулы?

Можно — и так делают. Идея: держать в контексте только имя+описание тула, а полную схему подгружать, когда модель захочет его вызвать. Но есть цена:

  • Навык раскрывается «бесплатно по числу витков»: это просто текст (.md), модель дочитала его и поехала дальше в том же ходу.
  • Тул так не выйдет: чтобы его вызвать, нужен машинно-точный tool_call{name, arguments}. Без схемы модель не знает параметров → сначала отдельным витком просит схему, и только следующим витком вызывает. +1 раунд на каждый вызов.

Компромисс (точка перелома):

  • мало тулов → держим схемы всегда: вызов в один виток, ленивость лишь вредит;
  • сотни тулов → полные схемы = десятки тысяч токенов «оглавления» в каждом запросе; выгодно проглотить +1 раунд на подгрузку, чтобы не платить контекстом постоянно.

Общий принцип (он же у навыков, у read_file с offset/limit, у субагентов): раскрывать по требованию стоит тогда, когда постоянная стоимость «держать всё» перевешивает разовую стоимость «подгрузить, когда нужно». Живой пример — deferred tools в Claude Code: сотни инструментов видны лишь по имени, схема подтягивается поиском по запросу.

Развёрнутое пояснение: откуда модель вообще знает, что нужный навык существует

Метаданные уровня 0 — не просто «список имён», это единственное, на основании чего модель принимает решение вызвать load_skill. Значит качество однострочного описания навыка («когда применять») напрямую определяет, найдёт ли модель нужный навык вовремя. Слишком общее описание («помогает с файлами») теряется среди похожих; слишком узкое — не сработает на слегка другой формулировке задачи пользователя. Это тот же компромисс, что при написании докстроки функции для другого разработчика: должно быть достаточно конкретно, чтобы узнать нужный момент, и достаточно кратко, чтобы не стоить дорого держать в контексте постоянно. Постепенное раскрытие снимает проблему объёма, но не снимает проблему точной формулировки на уровне 0 — она остаётся человеческой (или модельной) задачей проектирования навыка.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: При 500 тулов размер контекста большой или маленький? А при 500 навыках?

О: У тулов — огромный (полные схемы на каждом витке). У навыков — маленький: висят только метаданные, тело грузится по требованию. Потому навыки и сделали не как тулы.

В: Почему схему тула нельзя спрятать, а навык можно?

О: Тул вызывает обвязка — ей нужны точные параметры заранее. Навык применяет сама модель — до вызова ей достаточно знать, что он есть и когда нужен.

В: Раскрытый навык идёт в системный промпт или сообщением в messages[]? Почему?

О: Сообщением роли tool в messages[] — как результат любого тула. В системный промпт нельзя: он ушёл первым и неизменяем, а нужда в навыке выясняется в середине диалога.

6. Трёхуровневое сжатие контекста

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«По сути там просто подсчёт: если контекст занят на 92% — выполняем сжатие, просим модель обобщить всё, а недавние сообщения сохраняются дословно, потом записываем в .md и подгружаем при старте; и при каждом вызове модели проверяем заполнение?»

Главная мысль: не один скачок, а лесенка из трёх рычагов

Уровни отличаются масштабом того, что сжимаем, и как часто срабатывают — от мелкого/частого/дешёвого к крупному/редкому/дорогому. Смысл: не бить кувалдой там, где хватает скальпеля.

  1. Уровень 1 — обрезка отдельного результата инструмента (на лету, постоянно). Не весь диалог, а одно раздутое tool-сообщение. Прочитали файл на 5000 строк — через несколько витков он не нужен, тело заменяют заглушкой [результат read_file свёрнут]. Самые жирные и старые результаты выкидываются первыми, ещё задолго до порога.
  2. Уровень 2 — суммаризация куска старых ходов (промежуточный). Берём пролёт старых витков (давно закрытая подзадача) и схлопываем в короткий абзац «что было и чем кончилось». Локальный пересказ участка, а не всего.
  3. Уровень 3 — полный compaction (редкий, по порогу ~92%). Уперлись в потолок окна → отдельным API-вызовом модель пересказывает весь диалог, кроме свежего хвоста → новый контекст = [большой summary] + дословный хвост.

За счёт уровня 1 (по дешёвке выкидываем стухшие results) до дорогого уровня 3 часто вообще не доходит.

Почему свежий хвост — дословно

В последних сообщениях — горячее рассуждение: какой файл сейчас правим, что только что упало, что модель решила делать дальше. Пересказ их огрубляет → агент теряет нить на полушаге. Поэтому хвост оставляем буквально, а суммируем только старое.

Почему порог ~92%, а не 100%

Один вызов сжатия = инпут + аутпут в одном бюджете окна:

  • инпут — почти полный контекст (~92%): мы просим пересказать всё, что есть;
  • аутпут — сам summary, и он тоже ест из того же лимита окна.

Если ждать до 100% инпута — модели некуда генерировать summary, вызов физически не выполнится. Плюс тот обычный виток, который упёрся в потолок, должен успеть дописать ответ. Поэтому ~8% — резерв на (а) дать текущему витку дописаться и (б) дать сжимающему вызову место под его выход.

Тонкость о ролях токенов: на своём вызове summary — это выход (ест место). А на следующем шаге он перестаёт быть выходом и становится крошечным входом для всех будущих витков — окно резко освобождается. Один и тот же текст в разные моменты играет разные роли.

Сжатие ≠ память между сессиями (две роли — иногда один файл)

  • Compaction (все три уровня) живёт внутри одной живой сессии, в messages[]. Цель — влезть в окно сейчас и продолжить.
  • Память между сессиями (MEMORY.md / CLAUDE.md / .agent_memory.md) переживает конец сессии и грузится при старте новой. Цель — не забыть между запусками.

В этой статье обе роли намеренно склеены одним артефактом: summary уровня 3 пишется в .agent_memory.md, и следующая сессия стартует не с нуля. Но концептуально это всё равно две разные роли (сжать здесь и сейчас ≠ запомнить на завтра), просто реализованные одной сущностью.

Где живёт исходное содержимое после сжатия

Окно — это рабочий стол, а не сейф. Выбросить тело старого read_file безопасно: данные лежат на диске. Понадобилось снова — агент просто повторно вызывает read_file тот же путь. Контекст — не единственный носитель данных.

Развёрнутое пояснение: чем платит агент за неудачное сжатие

Стоит явно назвать риск, который лесенка из трёх уровней старается минимизировать, но не убирает полностью: суммаризация — это тоже вызов модели, а значит тоже может ошибиться или упустить деталь, которая позже окажется важной («агент решил, что миграция базы данных уже не актуальна для контекста, и обобщил её в одну строку — а через десять витков выяснилось, что именно там была причина текущего бага»). В отличие от уровня 1 (выкинутый read_file всегда можно перечитать с диска — потеря обратима), потеря на уровне 3 необратима: исходные ходы разговора уже не восстановить, summary — это единственное, что осталось. Поэтому свежий хвост оставляют дословно не только ради «горячего рассуждения» — это ещё и явная граница ответственности: суммаризатор рискует только старым, до чего разговор дошёл раньше, и никогда не трогает то, что ещё не устоялось.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: На уровне 1 выкинули тело старого read_file. А если файл снова понадобится — данные же удалены?

О: Не катастрофа — агент повторно вызовет read_file по тому же пути. Окно — рабочий стол, диск — сейф; стух → смахнули, понадобилось → взял с полки опять.

В: Почему порог ~92%, а не 100%?

О: Вызов сжатия тратит окно и на инпут (≈весь контекст), и на аутпут (сам summary). На 100% инпута выходу некуда генерироваться. Резерв ~8% — на дозапись текущего витка и на выход суммаризатора. На следующем шаге summary становится крошечным входом и освобождает окно.

В: Compaction и память между сессиями — одно и то же?

О: Нет, две роли: сжатие — влезть в окно живой сессии; память — пережить конец сессии. В этой статье склеены одним файлом .agent_memory.md, но роли разные.

7. Файловый граф зависимостей задач

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«Если TodoWrite — это простой список дел, то граф — полноценный задачник как рабочий процесс: исходная задача дробится на 100 подзадач с зависимостями друг от друга. Граф хранится в .agent_tasks.json и переживает падения процесса, перезапуски сессии и перезагрузки машины. Как его вообще представить?»

Главная мысль: шаг вперёд по двум осям сразу

Граф задач — это TodoWrite, прокачанный по двум осям одновременно: плоский список → граф (узлы + рёбра) и память процесса → файл на диске.

  • Узел = задача. Ребро = зависимость «B не может стартовать, пока не закрыта A».
  • Получается направленный граф: A ──► B ──► D, C ──► D («D ждёт B; B ждёт A и C»).

Что граф даёт, чего список не может

Список говорит только «что есть и что сделано». Граф добавляет слово «готова»: по рёбрам можно вычислить разблокированные задачи — те, у кого все зависимости уже закрыты. Их берём в работу, потом пересчитываем, что разблокировалось дальше. Это и есть топологический порядок выполнения.

Плоский список такой порядок выразить не может — в нём нет рёбер, нечего пересчитывать. Поэтому для сложной работы (десятки подзадач с зависимостями) нужен граф, а для простой линейной — хватает TodoWrite. Они сосуществуют, граф не убивает TodoWrite.

Кто считает «готова» и кто пишет файл — это КОД, не модель

Ловушка: кажется, что «модель сама помечает, что сделала, глядя в контекст». На деле два действия разделены:

  • Смысл («задача 7 закрыта») решает модель — вызовом инструмента update_task(id=7, status="done").
  • Запись в .agent_tasks.json и вычисление готовых задач по рёбрам делает код, а не модель.

Следствие: весь граф из 100 узлов не обязан лежать в контексте. Источник правды — файл на диске; код читает его и подаёт модели только нужный срез (готовые задачи + соседи). Та же мысль, что в сжатии: окно — стол, диск — сейф.

Рестарт после падения

Правильная картинка — не «модель вспоминает по контексту», а «код восстанавливает состояние из файла»:

  1. Процесс упал на середине большой работы, его перезапустили.
  2. Код при старте читает .agent_tasks.json: видит «1–40 done, 41 in_progress, остальное pending».
  3. Пересчитывает готовые задачи по рёбрам и даёт модели продолжить с места обрыва — не с нуля.

Зазор «сделал — но не успел пометить»

Агент выполнил задачу 41, и ровно в этот момент процесс упал — вызвать update_task(41, done) не успел. После рестарта код прочитает «41 не выполнена» и запустит её повторно.

Две идеи, образующие надёжный граф задач:

  • Идемпотентность — задачу проектируют так, чтобы повтор был безвреден («создать файл, если его нет», а не «создать файл»). Тогда повторный запуск 41-й максимум стоит лишний вызов, но ничего не ломает.
  • Atomic write файла + мелкие задачи/чекпоинты — чтобы сам .agent_tasks.json не побился при падении на середине записи, а потеря работы была дешёвой.

Зазор между «сделал работу» и «записал, что сделал» нельзя закрыть полностью — это фундаментально. Любая метка прогресса — это тоже запись, то есть ещё одно действие, и упасть можно в новом зазоре. Дробление задачи делает зазор уже, но не убирает его. С дырой не борются — с ней живут, делая последствия дешёвыми (идемпотентность + мелкие шаги).

Развёрнутое пояснение: почему это не то же самое, что просто «сохраняться почаще»

Кажется, что зазор «сделал — не успел пометить» можно сузить до нуля, просто сохраняя состояние после каждого микрошага. Отчасти это верно (дробление действительно сужает зазор), но полностью убрать его невозможно даже теоретически — сама запись прогресса требует времени, за которое тоже можно упасть. Это не инженерная недоработка, а версия той же проблемы, что известна в распределённых системах под именем «проблема двух армий»: невозможно достоверно синхронизировать два события (действие и подтверждение о нём) без риска, что связь оборвётся ровно между ними. Отсюда вывод конспекта: раз зазор принципиально неустраним, единственный рабочий рецепт — не пытаться его закрыть, а сделать последствия его срабатывания дешёвыми (идемпотентность). Это смещение цели с «предотвратить» на «сделать безопасным при повторении» — паттерн, который дальше встретится ещё раз в теме про снимки файлов.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: Что граф даёт, чего плоский список не может?

О: Порядок выполнения. По рёбрам код вычисляет готовые (разблокированные) задачи — у кого зависимости закрыты, — выполняет их, пересчитывает, что разблокировалось дальше. В списке рёбер нет, такого пересчёта быть не может.

В: Кто помечает задачу done и кто считает «готова»?

О: Смысл «done» решает модель вызовом update_task; саму запись в файл и вычисление готовых задач по рёбрам делает код. Весь граф в контекст не тащим — файл это источник правды, модели идёт только срез.

В: Процесс упал между «сделал 41» и «пометил 41 done». Что будет и можно ли исключить?

О: Код прочитает «41 не сделана» и запустит повторно. Полностью исключить нельзя — зазор между действием и его записью всегда есть. Спасают идемпотентность (повтор безвреден) и мелкие задачи + atomic write (потеря дешёвая).

8. Фоновые задачи (асинхронная очередь h2A)

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«Не совсем понятны фоновые задачи. Как они работают? Агент их создаёт? Как понимает, когда нужны? И что значит "он не сидит без дела и что-то делает параллельно"?»

Главная мысль: разъять две вещи, которые обычно склеены

Есть цикл рассуждения (agent loop) и выполнение операции (I/O — тесты, компиляция, миграция). Обычно они склеены: код вызвал инструмент и блокируется на 45 секунд, весь агент стоит. Механизм h2A эту склейку разрывает.

  1. Модель просит пустить долгую операцию в фон (вызов инструмента с флагом фона).
  2. Код не ждёт: стартует операцию в отдельном daemon-потоке и сразу возвращает модели «задача запущена в фоне». Ход завершён.
  3. Модель продолжает другие ходы: планирует, читает файлы, запускает ещё инструменты. Главный цикл не заблокирован на I/O.
  4. Фоновый поток закончил → кладёт результат в общую очередь уведомлений.
  5. Цикл после каждого хода заглядывает в очередь и вставляет готовое как user-сообщение. На ближайшем ходу модель реагирует.

Что параллельно, а что нет

Модель работает ходами: один вызов = один ход, между ходами она «спит». Параллельны тут I/O операции и рассуждение модели над другими шагами — а НЕ два рассуждения сразу. Результат подхватывается не «отдельным звонком через 45 секунд», а на ближайшей границе хода.

Почему уведомление вставляют как user-сообщение

Результат обязан попасть в контекст, иначе модель его никогда не увидит (в очереди он лежит «немой»). Именно роль user, потому что:

  • system prompt фиксированный и кешируется, его не трогают на каждом ходу;
  • сообщение роли user — это ровно то, на что модель по своей природе реагирует следующим ходом. Уведомление маскируют под user-сообщение, чтобы оно само вызвало реакцию.

«Вброс сразу, а не через очередь» — ложное противопоставление

Модель пошаговая и непрерываема: видит что-либо только на границе хода и только как сообщение в messages[]. Другого канала нет. Поэтому «сразу увидела» и «через очередь» — не противоположности: очередь и есть доставка к ближайшей границе хода, быстрее физически не бывает. Вброшенное пользователем сообщение идёт тем же путём — становится user-сообщением и читается на ближайшем ходу.

Что фон реально даёт для отзывчивости — не «минует очередь», а держит цикл живым. При блокирующей операции нить цикла застряла внутри шага «исполнить инструмент» (await/wait()) и не доходит до «собрать messages[] и позвать модель» — вброс лежит, брать некому. Фон уводит операцию в другой поток → нить свободна → ход случается сейчас, а не через 45 секунд.

Кто решает: фон или нет

Решает модель, обвязка не решает и не проверяет. Механизм тот же, что у любого выбора инструмента: по описанию схемы (отдельный run_in_background или флаг background=true) + нуджу в системном промпте. Критерий в описании — «для долгих операций, результат которых не нужен для следующего шага». Ошибётся (пустит в фон нужное немедленно) — не поломка, а мягкая неэффективность: окажется в «делать нечего, жду». Обвязка даёт возможность, смысл выбирает модель — как везде.

Когда фон бесполезен

Выигрыш от фона = ровно тот объём независимой работы, что есть на руках. Если весь план упёрся в результат операции (следующий шаг зависит от вывода теста), модель нечем занять — фон ничего не даёт, остаётся только ждать. Плюс фона: пока операция крутится, агент остаётся отзывчивым — можно вбросить новое сообщение, и модель ответит, не дожидаясь конца операции.

Развёрнутое пояснение: даемон-поток — не бесплатный трюк

Слово «daemon» в описании фонового потока не случайно — это конкретное системное свойство: если основной процесс (агентный цикл) завершится, даемон-поток обрывается вместе с ним, не дожидаясь своего конца. Это осознанный выбор, а не недосмотр: агент не обязан «дожидаться» фоновых задач при аварийном завершении сессии, иначе одна забытая долгая операция могла бы держать процесс живым бесконечно. Оборотная сторона: если фоновая задача успела начать необратимое действие (например, начала запись в файл) и процесс упал раньше её конца, результат может остаться в промежуточном состоянии — здесь снова та же тема «зазора», что и в графе задач: асинхронность и параллелизм не отменяют проблему незавершённых действий, они лишь переносят её в новое место.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: Модель работает ходами и между ними «спит». В каком смысле она работает, пока крутятся тесты?

О: Параллельно 45 секундам идёт рассуждение модели над ДРУГИМИ шагами (не два рассуждения сразу — параллельны I/O и мышление). Результат подхватывается на ближайшей границе хода.

В: Почему результат вставляют как user-сообщение, а не в system prompt / просто в очередь?

О: Он должен быть в контексте, иначе модель не увидит (очередь молчит). system prompt фиксирован и кешируется; user-сообщение — то, на что модель реагирует следующим ходом.

В: Когда фон бесполезен?

О: Когда модели больше нечего делать — весь следующий шаг зависит от результата операции. Нет независимой работы → нет выигрыша.

В: Foreground-цикл застрял на 45-секундной операции, я вбрасываю сообщение. Почему модель не может его прочитать?

О: Нить цикла заблокирована внутри шага «исполнить инструмент» (await/wait), не доходит до «собрать messages[] и позвать модель». Вброс лежит, но следующий ход не начнётся, пока поток не вернётся. Фон уводит операцию в другой поток — нить свободна, ход случается сразу.

В: Как модель понимает, что задачу пустить фоном, а что нет?

О: Сама — по описанию инструмента (отдельный run_in_background или флаг background=true) + нуджу в промпте, критерий «результат не нужен для следующего шага». Обвязка не решает и не проверяет; ошибка = мягкая неэффективность, не поломка.

9. Постоянные товарищи с JSONL-почтовыми ящиками

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«Подраздел полностью непонятен. В чём отличие от субагентов — в том, что они могут между собой взаимодействовать?»

Точка отсчёта: что такое субагент

Чтобы увидеть отличие, вспоминаем субагента (тема про изоляцию): родитель зовёт run_subagent(task) → это как обычный вызов функции: родитель блокируется, ждёт единственный return-строку. У субагента свой изолированный messages[], а после возврата этот контекст выбрасывается. Итог: субагент односторонний, одноразовый, безымянный.

Главная мысль: четыре оси, по которым товарищ ≠ субагент

«Взаимодействуют между собой» — верно, но это лишь одна ось (топология). Названия «постоянные» и «почтовые ящики» кивают ещё на память и адрес.

  • Топология. Субагент — строго вниз (родитель→ребёнок), наверх один раз. Товарищи — любой↔любой, равноправные пиры.
  • Память. У субагента messages[] стирается после return (холодный старт каждый раз). Ящик — .jsonl-файл, куда сообщения дописываются и который переживает сессии. Контекст сохраняется, а не отбрасывается.
  • Адрес/имя. Субагент анонимен — родитель держит на него указатель. У товарища есть ящик по адресу, поэтому ему может написать кто угодно, а не только «родитель».
  • Механика обмена. Спавн субагента — синхронный вызов-и-блокировка. Ящик — асинхронный: дописал строку в файл и пошёл дальше, адресат вычитает на своём витке. Это та же логика доставки, что у h2A фоновых задач, только очередь персональная, двусторонняя и персистентная.

Цена постоянных товарищей

Почему не делать всё на них и не выкинуть субагентов — у мощи есть плата, которой у дешёвого одноразового субагента нет:

  • Раздувание контекста. Ящик-.jsonl не стирается и лезет в messages[] при каждом чтении → контекст пухнет, нужно сжимать/чистить.
  • Стоимость. Субагент отработал и умер — освободил память и деньги. Несколько постоянных товарищей — это несколько параллельно живущих полноценных циклов, постоянный расход токенов.
  • Дедлоки. A ждёт письма от B, B ждёт от A → взаимная блокировка, никто не двигается. Плата за асинхронность и равноправие.

Развёрнутое пояснение: почему «постоянный» не значит «всегда включённый»

Легко представить товарища как процесс, который непрерывно работает в фоне, ожидая писем. На деле «постоянство» относится к ящику (файлу, который переживает сессии), а не обязательно к живому процессу товарища. Практичная модель: товарищ может «спать» между обращениями — его цикл не крутится, пока в ящике ничего нет, — а просыпаться (запускать свой мастер-цикл заново, но с загруженной историей ящика в контексте) только когда пришло письмо. Это меняет экономику: держать десяток товарищей «на всякий случай» не значит держать десяток постоянно работающих (и оплачиваемых) циклов — можно держать десяток спящих ящиков и поднимать процесс только под конкретное сообщение, теряя немного латентности, но не платя за простой.

Контрвопросы (проверка понимания)

В: Может ли субагент во время работы сам переспросить родителя и продолжить с учётом ответа?

О: Нет. Связь односторонняя: ушёл вниз — вернулся наверх один раз. Диалог «туда-обратно» — это уже про товарищей с ящиками.

В: Что даёт «почтовый ящик», чего не может субагент с исчезающим messages[]?

О: Сохранение контекста между обращениями и сессиями — накопленная история, а не холодный старт. Это и есть слово «постоянный» в названии.

В: Ящик — синхронно или асинхронно, и на что похоже?

О: Асинхронно (дописал и ушёл, читают на своём витке) — та же механика, что у фоновых задач h2A, только очередь персональная, двусторонняя и переживает сессии.

В: Почему тогда не заменить субагентов товарищами полностью?

О: Цена: раздувание контекста от несохраняемого файла + стоимость нескольких живых циклов + риск взаимных блокировок. Субагент дёшев именно потому, что одноразовый и умирает.

10. FSM-протокол командного общения

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«По сути это просто машина состояний, которая применяется к агентам?»

Литерально — да, но зачем

FSM (finite state machine, конечный автомат) действительно «машина состояний, применённая к агентам». Но это как «фоновые задачи — это просто поток»: механизм назван, а смысл — нет. FSM появляется ровно поверх прошлой темы: постоянные товарищи с ящиками дали канал, но без правил — отсюда хаос переписки, потеря писем, невидимые дедлоки. FSM — это дисциплина поверх ящиков.

Главная мысль: состояние — это структура, а не смысл

У каждого агента в любой момент есть явное состояние (IDLE / WORKING / WAITING_REVIEW / BLOCKED / DONE), а в каждом состоянии разрешён лишь определённый набор входящих сообщений (переходов). Ярлык состояния не надо «вчитывать», как текст письма, — его достаточно сравнить. Значит, протокол обслуживает дешёвая детерминированная обвязка, а не модель. Та же сквозная логика, что «цикл выходит по наличию tool_calls, а не по смыслу ответа».

Что это лечит

  • Недопустимое сообщение не теряется и не обрабатывается «как попало». Пришло письмо, которого текущее состояние не ждёт (B в WAITING_REVIEW, а прилетело не ревью) → это недопустимый переход: FSM держит письмо в очереди до состояния, где оно допустимо, и тогда отвечает. На одно письмо в одном состоянии всегда один и тот же исход — детерминизм вместо произвола голого ящика.
  • Дедлоки становятся видимыми. A в WAITING_ON_B, B в WAITING_ON_A. В голом ящике оба висели бы молча вечно. Раз состояния — читаемые ярлыки, а обвязка умеет на них смотреть, она замечает цикл зависимостей и может его разорвать/сообщить.
  • Читаемость команды. По состояниям видно, кто чем занят и кто на что ждёт, — без вчитывания в переписку.

Почему это может делать тупая обвязка (крух темы)

Состояние — структура. Проверить «допустим ли переход» = сравнить ярлыки, а не понять смысл. Поэтому весь протокол вешается на детерминированный код: он очередит недопустимые письма, видит тупики, делает поведение предсказуемым. Обвязка даёт дисциплину, модель занимается содержанием.

Развёрнутое пояснение: кто определяет саму диаграмму состояний

Важно разделить два разных вопроса: кто проверяет переходы (обвязка, дёшево и детерминированно — уже разобрано) и кто придумывает набор состояний и разрешённых переходов между ними в принципе. Второе — не автоматика, это дизайн-решение, которое принимает человек, проектирующий систему, заранее, до того как агенты вообще начали работать (аналогично тому, как permissions.yaml из темы 15 — тоже заранее написанная человеком политика, которую код лишь исполняет). Если набор состояний спроектирован плохо — например, не хватает промежуточного состояния для частого сценария — FSM начнёт с завидной регулярностью отклонять законные сообщения как «недопустимые переходы». Надёжность протокола связана с надёжностью проверки лишь частично; она так же сильно зависит от того, насколько полно диаграмма состояний покрыла реальные сценарии общения агентов.

Контрвопросы (проверка понимания)

В: Кто способен обеспечивать FSM-протокол — умная модель или обвязка, и почему это плюс?

О: Обвязка. Состояние — структура, а не смысл; сравнить ярлык дёшево и детерминированно. Не нужно гонять координацию через дорогую модель.

В: B в WAITING_REVIEW, прилетело не-ревью письмо. Что с ним?

О: Это недопустимый переход — не выбрасывается молча, а очередится/обрабатывается предсказуемо до состояния, где такое сообщение допустимо. Ключ — детерминизм, а не «блокировка».

В: Как FSM ловит дедлок A↔B, который голый ящик не мог?

О: Состояния читаемы, обвязка смотрит на них и видит взаимный цикл зависимостей WAITING_ON. В свободном ящике оба просто молча висели бы вечно.

11. Автономное взятие задач агентами

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«По сути это дополнение FSM? Просто агенты читают общий граф задач и помечают, что вот я взял эту задачу.»

Литерально — нет, это другая ось

Это не про FSM. FSM — дисциплина общения между агентами (кто кому пишет, какое состояние какое письмо принимает). Здесь общения между агентами нет вовсе: никто никому не шлёт письмо. Связь тут с графом задач — тем самым durable-графом, где у задач есть зависимости и статусы (pending / готова / выполняется).

Push против Pull — кто решает, кто делает задачу

Push (наш координатор): начальник адресует задачу конкретному товарищу — «ты, arch, делай вот это». Решает, кто исполнитель, — координатор.

Pull (автономное взятие): задачи лежат в общей куче, никто их не адресует. Каждый агент крутит непрерывный цикл сканирования графа, находит незаблокированную ожидающую задачу и сам берёт её себе. Решает, кто исполнитель, — сам агент, по факту «кто первый схватил».

Крух темы: зачем «атомарно через Lock»

Взятие задачи — это два действия: (1) прочитать граф → «задача №5 свободна, pending»; (2) записать → «ставлю in_progress, беру себе». Между ними есть щель.

Если два агента сканируют одновременно, оба успевают прочитать «№5 свободна» до того, как кто-то записал, — и оба берутся за одну задачу (в худшем случае затирают запись друг друга).

Имена ошибок и лекарство

  • Состояние гонки (race condition) — исход зависит от того, кто из потоков успел раньше влезть в щель между «прочитал» и «записал».
  • Потерянное обновление (lost update) — A прочитал старое значение и затёр то, что B успел записать в щель.
  • Атомарная операция — лекарство: threading.Lock склеивает «прочитать + записать» в одно неразрывное действие. Пока A держит замок, B ждёт у входа; A читает, ставит in_progress, отпускает — и только тогда входит B и видит задачу уже занятой. Lock не «запрещает читать» — он не даёт разорвать пару чтение-запись. Оттого в тексте слово «атомарно».

Развёрнутое пояснение: цена, которую платит pull-модель за простоту

Push (координатор раздаёт задачи) и pull (агенты сами разбирают из кучи) не эквивалентны по качеству распределения работы, даже если Lock полностью решает гонки. При push координатор может учитывать специализацию («это архитектурная задача — дай её агенту arch, а не первому свободному»); при pull побеждает буквально тот, кто первым успел просканировать граф — без учёта, подходит ли задача именно ему. Значит pull хорош там, где задачи взаимозаменяемы (любой агент одинаково справится), и плох там, где важна специализация — тогда придётся либо возвращаться к push, либо усложнять pull фильтром «беру только задачи из своей области». Lock решает проблему корректности (никто не возьмёт задачу дважды), но не решает проблему качества распределения — это отдельная, более тонкая забота.

Контрвопросы (проверка понимания)

В: Это дополнение FSM? Есть ли тут общение между агентами?

О: Нет. Общения между агентами нет, никто не шлёт письмо. Связь — с графом задач, а не с FSM.

В: Кто решает, какой агент делает задачу, при push и при pull?

О: При push — координатор адресует конкретному. При pull — агент сам берёт из общей кучи, «кто первый».

В: Где прячется беда без Lock и что Lock делает?

О: В щели между «прочитал pending» и «записал in_progress» — второй агент успевает вклиниться и взять ту же задачу (race condition / lost update). Lock склеивает чтение и запись в атомарное действие: пока один держит замок, другой ждёт и потом видит задачу уже занятой.

12. git worktree — изоляция параллельной правки файлов

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«Что такое git worktree? В статье упоминается.»

Что это по сути

Обычно: один репозиторий = одна рабочая папка, одна ветка за раз. git checkout other подменяет файлы в той же папке — две ветки одновременно «в руках» не подержать. git worktree снимает это ограничение: у одного репозитория несколько рабочих папок сразу, каждая на своей ветке.

myrepo/            ← ветка main
myrepo-feature-a/  ← ветка feature-a
myrepo-feature-b/  ← ветка feature-b

Общая .git-история одна (коммиты/объекты не копируются), но каждая папка — независимый чекаут: свои файлы на диске. Правка в A не трогает файлы в B.

Зачем это в статье про агентов

Несколько агентов работают параллельно. Дать каждому свою worktree → каждый пишет в свою копию репозитория. «Два агента буквально пишут в разные файлы в разных каталогах — конфликт записи невозможен, потому что сами файлы разделены».

Ключ: «тот же файл» — это НЕ тот же файл

Интуиция «оба агента правят один a.py» — ложна. Это myrepo-A/a.py и myrepo-B/a.py — два разных файла на диске. Затирать строку друг друга физически нечем. Конфликт не исчез — он отложен до момента слияния веток.

Симметрия с темой 11 — два приёма против одного класса гонки

Тема 11 (взятие задач): общий ресурс неизбежен → защищаем Lock (сериализация, «по очереди к одному ресурсу»).

Worktree (правка файлов): общий ресурс не неизбежен → убираем его (копия каждому). Гонка не возникает не потому что караулим, а потому что делить нечего.

При слиянии выбирает человек, а не оркестратор

Git при merge не заставляет выбирать одно из двух по каждому файлу:

  • тронуты разные файлы / разные участки одного файла → сливается автоматически, берутся оба изменения;
  • изменена одна и та же строка по-разному → merge conflict, только он идёт на ручное разрешение.

Обвязка выносит человеку только пересечение, а не «весь результат каждого агента» — обычно это маленький кусок.

Почему человеку, а не модели-оркестратору. Merge conflict = два осмысленных, намеренных изменения одного места (не «мусор против правильного»). Тот же сквозной принцип: код держит структуру (git надёжно находит пересечение), но смысл — «какой из двух вариантов нужен продукту» — код решить не может. Плюс два признака, делающие ошибку тут особенно дорогой:

  • Необратимость (тихая): неверный выбор молча смёржится, всё «зелёное»; человек заметит через несколько шагов, когда сверху уже наросло и откат дорог.
  • Владение решением: полную картину нужного варианта держит только человек — ни git (видит структуру), ни отдельный агент (у него узкий кусок).

В реальном Claude Code экрана «выбери привет или пока» не видно, потому что subagent'ы обычно правят разные файлы и отдают результат родителю, который синтезирует; до git-merge между ветками агентов в стандартном флоу дело обычно не доходит. Worktree-схема с веткой на агента — продвинутый паттерн оркестрации, а не поведение по умолчанию.

Развёрнутое пояснение: почему нельзя просто «взять оба варианта»

Интуитивная мысль при конфликте — раз оба изменения намеренные, почему бы не сохранить оба (например, оставить обе версии строки друг за другом)? Иногда это работает (два независимых импорта в начале файла), но чаще ломает саму суть правки: если один агент переписал функцию на новую сигнатуру, а другой — независимо поправил старую версию той же функции, «оба варианта одновременно» — это не рабочий компромисс, а синтаксически или логически несовместимый код. Git физически умеет вставить оба фрагмента текста, но не умеет проверить, останется ли программа рабочей. Отсюда конфликт и выносится человеку не как формальность, а потому что только человек может решить, что «взять оба» здесь не вариант, и синтезировать третье, работающее решение.

Контрвопросы (проверка понимания)

В: Два агента «правят один a.py» — как получается, что конфликт записи невозможен?

О: Это не один файл, а две копии в разных worktree (myrepo-A/a.py и myrepo-B/a.py). Затирать нечего; конфликт лишь отложен до слияния.

В: Почему для задач выбрали Lock, а для файлов — worktree?

О: Один класс гонки, два приёма: где общий ресурс неизбежен — сериализуем Lock'ом; где его можно убрать — даём копию каждому, и делить становится нечего.

В: Почему пересечение выносят человеку, а не дают оркестратору выбрать строку?

О: Конфликт — два осмысленных намерения; выбор требует полной картины продукта (она есть только у человека), а ошибка тихо необратима — заметят поздно, когда откат дорог. Код находит пересечение (структура), смысл решает человек.

Слабая сторона? — где статья расходится с реальным Claude Code

Фактическая поправка: git worktree в Claude Code есть — можно вручную запустить несколько экземпляров, каждый в своей worktree на своей ветке, и пилить фичи параллельно. Чего нет «из коробки» — это автоматической обвязки, которая сама раскидывает subagent'ов по worktree, сравнивает ветки и выносит пересечение на ревью. Сам инструмент доступен, автоматики нет.

Это не дефект, а другая точка, где гасят гонку правок:

  • Модель статьи (ветки + merge): агенты работают вслепую друг к другу, конфликт всплывает в конце, при слиянии — постфактум, отдельным шагом ревью.
  • Модель Claude Code (родитель раздаёт файлы): оркестратор заранее нарезает работу так, чтобы агенты не лезли в одни файлы. Конфликт предотвращается на входе, а не разруливается на выходе. Предотвратить дешевле, чем чинить постфактум — но цена в том, что родитель должен уметь хорошо нарезать.

Где это реально ограничение: когда много агентов должны глубоко и независимо переписывать один и тот же файл. Тогда push-нарезка «разные файлы разным» не работает (файл один), а автоматики «каждому ветку → merge» из коробки нет. Но это редкий и обычно нежелательный сценарий: пять агентов, рвущих один файл, — чаще признак плохой декомпозиции, чем слабости харнесса. Ярлык «слабо» справедлив только для этого узкого края.

Общая оговорка ко всему конспекту. Статья местами отходит от реальной реализации Claude Code — она даёт педагогическую модель, а не документацию продукта. Уже встречавшиеся расхождения: (1) автономное взятие задач через threading.Lock (тема 11) — в реальном CC subagent'ы это отдельные вызовы модели без общей памяти, поэтому и замка нет, и раздаёт задачи родитель (push); (2) worktree-на-агента с авто-merge (тема 12) — инструмент есть, автообвязки нет. Концепции статьи верны как принципы («координация через общий артефакт», «изоляция контекста», «код держит структуру, смысл — наверх»), но конкретные механизмы — один из возможных способов их воплотить, не обязательно тот, что внутри продукта.

13. Потоковая выдача токенов — меняется только потребление

✔ разобрано и закрыто

В Claude Code потоковая выдача — не опция, а режим по умолчанию: токены появляются в терминале по мере генерации. Разница между потоковым и блокирующим агентом — это разница между инструментом-соработником и пакетной задачей. На коротких ответах разрыв незаметен; но в длинной цепочке рассуждений с десятками вызовов инструментов блокирующий агент молчит минутами, а потоковый показывает свою работу в реальном времени.

Главная мысль: переход блокирующий → потоковый меняет только способ потребления ответа. Логика цикла та же. Логика диспетчеризации та же. Решение о выходе — по-прежнему по структуре ответа (есть tool_calls → продолжаем, нет → финал), а не по смыслу текста. Меняется лишь то, как из ответа достаётся цельный результат.

Почему «ничего ниже не ломается»

В потоке ответ приходит не готовым объектом, а осколками (дельтами): куски текста, куски вызовов инструментов. Но после того как поток собран обратно в цельный ход модели, вся нижележащая логика — проверка причины остановки, разбор tool_use, извлечение блоков — работает ровно как раньше. Поэтому потоковый и блокирующий цикл — это один и тот же цикл; отличается только слой «достать цельный ответ из потока».

Тонкость сборки (и ещё одно расхождение статья↔реализация). В примере статьи (Anthropic SDK) сборку осколков делает библиотечный метод — он блокируется до конца потока и возвращает полный объект той же структуры, что и обычный вызов. То есть склейка есть, но спрятана внутри готового метода. В OpenAI-совместимом протоколе (наш случай с DeepSeek) такого метода нет: поток отдаёт сырые дельты, и tool_calls приходят кусками в разных чанках — их надо склеивать вручную по индексу (имя и строку аргументов накапливать по слотам). Это самодельный аналог «собрать полный ответ». Тезис «меняется только потребление» это не нарушает, а подтверждает: склейка осколков и есть часть потребления.

Где во витке живёт склейка — и почему это не про раскрытие. Сборка осколков идёт после вызова модели (это разбор ответа). А набор инструментов tools[] собирается до вызова (прогрессивное раскрытие, тема 6). Разные концы витка → потоковая склейка и раскрытие инструментов не связаны: раскрытие могло бы вообще отсутствовать — фрагментацию в потоке всё равно пришлось бы склеивать.

Развёрнутое пояснение: что стриминг меняет для пользовательского восприятия, а не для логики

Стоит отделить два эффекта, которые легко смешать. Первый — технический (уже разобран): нижележащая логика цикла не меняется, меняется только сборка ответа из осколков. Второй — сугубо человеческий: пользователь, видящий текст по мере генерации, воспринимает систему как отзывчивую даже если суммарное время до финального ответа то же самое, что при блокирующем режиме. Это тот же принцип, что используют индикаторы прогресса в обычном ПО — не ускоряют работу, а меняют субъективное ощущение ожидания. Для агента с десятками вызовов инструментов это особенно ощутимо: пользователь видит «сейчас читаю файл X», «запускаю тесты» вместо голого молчания — и не гадает, завис агент или работает.

В: Противоречит ли ручная склейка tool_calls тезису «меняется только потребление»?

О: Нет, подтверждает. Склейка осколков — часть потребления ответа. В Anthropic SDK она спрятана в библиотечном методе, в OpenAI-совместимом её пишут руками. Логика ниже (выход по структуре, диспетчеризация) не меняется ни на строчку.

В: Склейка работает до вызова модели или после? Причём тут раскрытие инструментов?

О: После — это разбор ответа. tools[] же собирается до вызова. Разные концы витка, поэтому раскрытие к склейке отношения не имеет.

14. Снимки файлов и revert — обратимость правок

✔ разобрано и закрыто

Каждый write (и edit) незаметно сохраняет прежнее содержимое файла перед перезаписью. Если правка модели что-то ломает — revert возвращает исходное состояние одним вызовом.

«Это просто бэкап файла?» По механике — да, копия в сторонку. Но «просто» недооценивает три вещи вокруг копии:

  • Автоматизм. Снимок делает код обвязки при каждой записи, а не модель по своей воле. Модель может забыть, поспешить, сохранить копию не того файла — а страховка срабатывает всегда, потому что она в коде, а не в решении модели.
  • Дешёвая обратимость → смелость модели. Раз откат стоит один вызов, модель может действовать решительно (переписать файл целиком), не боясь всё молча сломать. Снимок — это то, что делает необратимое обратимым.
  • Кто решает откат. Сам revert — структура (тупое восстановление байтов). А когда откатывать («правка сломала сборку») — смысл, его решает модель или человек. Бэкап не решает — он лишь даёт кнопку.

Почему снимок делает код, а не модель (связь с темами 7 и 11)

Если бы снимок был отдельным вызовом модели («сначала сохрани копию, потом перезапиши»), возник бы зазор: модель могла бы перезаписать файл, не успев (или забыв) снять копию — и оригинал потерян навсегда. Это тот же зазор «сделал — не успел пометить» из темы 7 (durable-граф).

Лечение то же — атомарность. Раз снимок вшит в код самого write, снять-снимок и перезаписать неразрывны: физически нельзя затереть файл, не сняв снимок. Между ними нет щели, в которую что-то провалится при сбое. Тот же приём, что и в темах 7 и 11 — склеить пару действий в одно; разница лишь в средстве: там os.replace / threading.Lock, тут — просто порядок строк внутри одной функции обвязки.

Снимок — только на запись

Снимок нужен ровно в момент, когда старое содержимое вот-вот затрётся, — то есть перед write/edit. Чтение файла терять нечего, поэтому на read снимок не делается.

Нумерованный вывод строк делает точные правки возможными

Смежный механизм: read отдаёт строки с номерами (1 …, 2 …). Статья говорит: именно нумерация делает точные правки возможными. Но важно, как — если правка идёт по уникальному фрагменту (old_string, который должен встречаться ровно один раз), а не по номеру строки.

Номер — не адрес правки, а ориентир. Пусть строка return None встречается в файле трижды. Инструмент правки, получив только return None, откажется: непонятно, которое из трёх вхождений менять. Нумерация даёт модели два шага: (1) сориентироваться — увидеть, что нужное вхождение на строке 13, и что вокруг него; (2) собрать однозначный якорь — взять кусок побольше (с соседними строками), который встречается уже ровно один раз. Так номер даёт позицию и контекст, чтобы построить уникальный old_string, а сам инструмент правит по тексту, не по номеру. Без нумерации модель правит вслепую и легко зацепит не то вхождение.

А в реальном Claude Code бэкапы есть? Да, но зерно другое. У нас (и в статье) единица отката — одна запись: снимок на каждый write, откат пофайлово вызовом инструмента. В Claude Code это чекпоинты: /rewind (и двойной Esc) откатывает и файлы, и состояние разговора к более раннему ходу диалога — через команду UI, а не инструментом в руках модели. Суть общая — дешёвая обратимость правок, — но единица отката у них = шаг разговора, а не отдельный write. Ещё один случай «статья даёт педагогическую модель, продукт реализует по-своему» (см. общую оговорку в теме 12).

Нюанс «файла не было». Если модель создала новый файл, честный откат — вернуть к состоянию «не существовал», то есть удалить файл, а не оставить пустышку. Поэтому снимок различает «файл был с таким-то содержимым» и «файла не было» (отдельный маркер). А чтобы откатывать несколько последовательных правок по одной назад, снимки складываются в стек на каждый путь — это история undo, а не только «последнее хорошее».

Развёрнутое пояснение: снимки и compaction тянут в разные стороны

Стоит явно увидеть напряжение между этой темой и темой сжатия контекста: снимки живут на диске, а не в messages[], поэтому compaction (даже самый агрессивный, уровень 3) их не трогает — стек истории правок переживёт любое сжатие разговора. Это осознанная асимметрия: разговор — расходный ресурс, который можно безопасно ужимать, а история правок файлов — вид данных, для которого потеря немедленно означает «нечего восстанавливать». Отсюда практическое следствие: даже если сессия сожмётся до одного абзаца summary, откатить последнюю правку файла всё ещё можно — способность к revert не зависит от того, насколько «похудел» контекст разговора.

В: Обвязка просто делает бэкап файла?

О: По механике — да. Но суть не в самой копии, а в том, что её делает код автоматически перед каждой записью (модель бы забыла), что откат дёшев (значит модель смелее), и что решение «откатывать» остаётся за моделью/человеком, а не за бэкапом.

В: Почему снимок должен делать код, а не модель отдельным вызовом?

О: Иначе зазор как в теме 7: модель может перезаписать файл, не успев сохранить копию, — оригинал потерян. Вшитый в код снимок неразрывен с записью (атомарность), щели нет.

15. Разрешения через YAML-правила — политика живёт вне кода

✔ разобрано и закрыто

При первом запуске Claude Code спрашивает: работать в режиме авто-подтверждения или в режиме подтверждений. Этот выбор ложится на систему уровней: одни команды всегда идут молча, другие всегда просят явного «да», третьи блокируются наглухо независимо от контекста.

Три уровня, заданные конфигурацией

Политика описана в permissions.yaml тремя списками шаблонов (regex), проверяемых по порядку:

  • always_deny — блок без исключений, проверяется первым (rm -rf /, sudo, curl … | sh). Совпало → отказ, дальше не смотрим.
  • always_allow — тихий пропуск без уведомления (ls, read-only git: status/log/diff, --version).
  • ask_user — пауза и запрос подтверждения (rm, git commit/push, pip/npm install, доступ к .env).
  • По умолчанию (ни одно правило не совпало) — разрешить.

Почему порядок именно такой

Сначала deny, потом allow, потом ask — чтобы запрет был сильнее разрешения. Иначе rm -rf / мог бы «случайно» проскочить через более общий разрешающий шаблон. Первый матч выигрывает, поэтому самый жёсткий список стоит впереди.

Главная мысль: политика безопасности — в конфиге, а не в коде. Поменять то, что требует подтверждения, — это правка YAML-файла, а не перевыкладка кода. Проверка правил — обёртка (check_permission), которая срабатывает перед каждым вызовом инструмента, до его выполнения. Тот же повторяющийся принцип конспекта: код держит структуру (шлюз, порядок уровней), а конкретная политика — данные снаружи. Правила можно менять, не трогая механизм.

Как это выглядит в работе

Запрос «поставь requests, удали старый requirements.txt и собери новый» дал два запроса подтверждения — на pip install и на rm (оба из ask_user). А ls, read, pip freeze прошли молча — совпали с always_allow. Модель свободно делает безопасное и тормозит лишь там, где у действия есть реальные последствия.

Связь с «структура vs смысл». Шлюз разрешений — чистая структура: тупое сопоставление regex, никакого понимания намерения. Он не спрашивает модель «а ты уверена?» — он механически ловит опасный текст команды. Решение по существу (нажать «y») остаётся за человеком в режиме подтверждений. Обвязка лишь ставит на паузу в нужной точке.

Развёрнутое пояснение: слепое пятно regex-шлюза

Ровно потому что проверка чисто текстовая, у неё есть честная граница: она ловит опасную команду, но не опасный эффект, если он достигнут окольным путём. Классический пример — цепочка безобидных по отдельности действий (запись безобидной с виду строки в файл, который потом кем-то выполняется) может обойти шаблон, написанный под конкретный опасный вызов вроде rm -rf. Это не повод считать шлюз бесполезным — он честно закрывает большинство прямых случаев дёшево и без участия человека, — но иллюстрирует границу принципа «структура вместо смысла»: там, где опасность зависит от смысла цепочки действий, а не текста одной команды, чисто структурная защита предсказуемо слабее. Отсюда на практике regex-правила в permissions.yaml обычно дополняют, а не заменяют более широкие ограничения окружения (песочница, отдельный пользователь ОС, сетевые ограничения).

В: Почему always_deny проверяется первым, а не, скажем, ask_user?

О: Первый совпавший уровень выигрывает. Запрет должен быть сильнее разрешения, иначе опасная команда могла бы проскочить через более широкий allow-шаблон. Поэтому жёсткий блок стоит впереди.

В: Что даёт вынос политики в YAML вместо хардкода в коде?

О: Изменить, что требует подтверждения, — это правка конфига, а не развёртывание. Механизм (обёртка-шлюз перед каждым вызовом) неизменен, а политика — данные снаружи. Код держит структуру, политику решают отдельно.

16. Шина событий и хуки жизненного цикла — наблюдаемость вне цикла

✔ разобрано и закрыто

Команде нужно навесить на агента учёт стоимости, журнал аудита, свои шаги подтверждения, интеграцию с мониторингом. «Наивный» путь — вписать всё это внутрь агентного цикла. Тогда цикл распухает и превращается в свалку не связанной с ним логики, а его сердце (то, что легко сломать) правится ради каждой мелочи. Хуки решают ровно эту проблему: они придуманы, чтобы код цикла НЕ менять.

Главная мысль: наблюдаемость — структурное свойство обвязки, а не дополнение «после факта». Цикл не знает про логи, таймеры и статистику. Он лишь объявляет вслух каждый значимый момент именованным событием. Кто и как на это реагирует — уже не его забота. Тот же повторяющийся принцип конспекта: механизм зашит в код (цикл + шина), а поведение подключается снаружи (хуки) — как политика в теме 15 живёт в конфиге, а не в коде.

Три части механизма

  • Шина (EventBus) — простой диспетчер: словарь событие → список обработчиков. Два метода: on(event, handler) подписывает функцию, emit(event, **payload) «выстреливает» событием — проходит по всем подписчикам, зовёт каждый с полным payload и собирает их возвраты.
  • Хуки — обычные функции, ничего не знающие друг о друге. Один хук может висеть на нескольких событиях (сам внутри разбирает, что пришло), а на одно событие может быть подписано сразу несколько хуков. Связь «многие-ко-многим».
  • Цикл — только генерирует события вокруг каждого вызова инструмента: session_start в начале, pre_tool_use перед исполнением, post_tool_use после успеха, tool_error в except, session_end в finally (то есть гарантированно, даже при вылете).

Хук может заблокировать вызов

pre_tool_use — не просто уведомление. Цикл смотрит на возвраты хуков: хук, вернувший {"block": True}, запрещает запуск инструмента. Так, по словам статьи, «слой enforcement аккуратно накладывается поверх управления разрешениями». Прямая связь с темой 15: permissions-шлюз — встроенная политика, а хук pre_tool_use — точка расширения снаружи, не трогающая ни цикл, ни конфиг разрешений.

Отказоустойчивость: emit ловит исключение каждого хука. Упавший логгер не имеет права уронить агента — работа продолжается, просто без лога. Наблюдатель не убивает наблюдаемого.

Tool vs Hook — разные вещи

  • Инструмент вызывает сама модель — это её способность (прочитать файл, bash). Добавляется кодом.
  • Хук срабатывает автоматически, помимо воли модели — это правило/наблюдение вокруг того, что она и так делает. Вешается снаружи, без правки цикла.

Ключевое: модель не может забыть сработать хук, потому что решает не она, а обвязка. Поэтому то, что обязано случаться всегда (форматирование после правки, аудит, жёсткий запрет на .env), доверяют хуку, а не промпту.

В настоящем Claude Code

Статья строит EventBus с нуля, чтобы показать устройство. В реальном Claude Code тот же механизм уже встроен: хуки настраиваются не кодом, а в settings.json (команда /hooks — лишь мастер, редактирующий этот файл через меню). Отличие от статьи: там хук — Python-функция внутри процесса, а в Claude Code — внешняя shell-команда, которой на stdin приходит JSON события и которая может ответить (в т.ч. заблокировать вызов).

Развёрнутое пояснение: почему хук — внешний процесс, а не функция в памяти

То, что реальный Claude Code делает хук отдельной shell-командой (а не Python-функцией внутри процесса, как в учебной реализации статьи), — не случайная деталь, а следствие того, кто хук пишет. Пользовательский хук должен работать на любом языке, который умеет читать stdin и писать stdout/exit-код — Python, Bash, Node, скомпилированный бинарник. Заставить хук быть Python-функцией внутри процесса означало бы, что каждый, кто хочет что-то навесить на событие, обязан писать код на том же языке и в том же процессе, что и сам агент — тесная связка, ломающая ту самую изоляцию, ради которой хуки задуманы. Плата за это — накладные расходы на запуск отдельного процесса при каждом событии, но она принимается ради того, чтобы точка расширения оставалась языконезависимой и по-настоящему внешней.

В: Почему форматирование кода правильнее сделать хуком, а не попросить модель «всегда форматируй после правки» в промпте?

О: Промпт — это просьба, которую модель может забыть или проигнорировать. Хук на post_tool_use срабатывает гарантированно и всегда, потому что решает обвязка, а не модель. То, что обязано случаться каждый раз, нельзя доверять доброй воле модели.

В: «Под каждый хук надо менять код» — так ли это?

О: Наоборот. Хуки придуманы, чтобы код цикла НЕ трогать. Цикл и так генерирует события; ты лишь снаружи (в конфиге / через bus.on) говоришь, что на них делать. Кодом добавляют инструменты, а хуки вешаются вне цикла.

17. Сохранение, продолжение и ветвление сессий — история как данные

✔ разобрано и закрыто

Исходный вопрос

«Claude Code локально сохраняет каждое сообщение, вызов инструмента и результат. Что за REPL и правда ли сохраняется вообще всё — любой запрос модели, любой вызов тулов?»

Что такое REPL

REPL = Read–Eval–Print Loop — тот самый вечный цикл общения в терминале: читает твою строку → выполняет (гоняет мастер-цикл, зовёт модель и тулы) → печатает ответ → возвращается ждать следующую строку.

REPL-команды (/save, /resume, /branch) вводятся в то же приглашение, но НЕ идут в модель — это служебные строки, управляющие самим агентом, а не сообщение для ИИ.

Главная мысль: сохранение — это просто дамп того, что уже есть

Никакого отдельного «журнала» или шпиона не заводят. Вся история и так живёт в одном списке messages: твои строки, ответы-мысли модели, её tool_calls, результаты role:"tool". Модель без этого списка слепа — на каждом витке мы шлём ей его целиком. Поэтому сохранить сессию = сериализовать messages на диск (условно json.dump), а продолжить = загрузить обратно.

Отсюда прямой ответ на «сохраняется ли вообще всё»: да — но не потому что где-то стоит логгер, а потому что по определению всё уже собрано в одной структуре, иначе модель не могла бы работать.

Три команды, а не одна

  • save — сброс messages в файл (перезапись текущей ветки).
  • resume — загрузка файла обратно в messages, цикл крутится дальше, будто перерыва не было.
  • branch — загрузка старого состояния, но дальнейшие сообщения пишутся уже в новый файл / под новым id, а исходный остаётся нетронутым.

Ветвление = дерево вместо прямой линии

Ключевое слово — не перезаписывать. Загрузил состояние на 40-м сообщении, наделал 10 «не туда», захотел вернуться и попробовать иначе, не потеряв первый вариант → от 40-го расходятся две ветки, обе целы. Простой парой «прочитал / перезаписал» так нельзя: перезапись затирает то, от чего ты ответвился.

Важное: ветвление ничего не требует от самого агентного цикла. Цикл как крутил один messages, так и крутит — он не знает, что он «ветка №2». Вся магия в том, куда сохраняем и откуда грузим. Это команды СНАРУЖИ цикла — ровно как хуки (тема 16) висели снаружи, не трогая loop.

Почему это REPL-команды, а не tools модели

save/resume/branch — операции над самим разговором, а модель живёт внутри него. Просить её «сохрани сессию» — всё равно что просить героя книги переплести книгу.

  • Уровень. Команды работают с messages как с объектом целиком; модель видит лишь содержимое списка, а не сам список как файл.
  • Контроль у человека. Ветвление — твоё решение «попробовать иначе». Тот же шов, что и везде: структура/обвязка исполняет, смысл решает человек сверху.
  • Надёжность. Tool модель может забыть, вызвать не вовремя, зациклить. Команда, которую жмёшь ты, срабатывает детерминированно.

Развёрнутое пояснение: чем сессия отличается от снимков файлов

Легко перепутать сохранение сессии (тема 17) со снимками файлов (тема 14) — обе темы про «сохранить состояние и вернуться назад». Разница в том, что именно является состоянием. Снимок фиксирует содержимое одного файла на диске в момент до перезаписи — это состояние мира, с которым работает агент. Сохранение сессии фиксирует messages[] — состояние разговора, то есть что агент думал и делал. Они не синхронизированы друг с другом: можно откатить разговор к 40-му сообщению через /resume, но файлы на диске при этом останутся в том состоянии, в которое их привели уже выполненные с тех пор правки — откат разговора не откатывает файлы автоматически. Если нужно вернуть согласованную картину «и разговор, и файлы к моменту X», это два независимых действия, а не одно.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: Сохранение — это новый механизм логирования?

О: Нет, просто запись на диск того, что уже лежит в структуре messages. Дамп списка = сохранённая сессия.

В: Зачем branch, если есть save/resume?

О: Чтобы увести историю в отдельную ветку, не затирая исходную. Загрузка старого состояния + запись в новый файл → история становится деревом, оба варианта целы.

В: Почему это REPL-команды, а не tools?

О: Это операции над самим разговором, а модель внутри него — контейнером управлять должна обвязка/человек, детерминированно, а не модель через tool call.

18. Асинхронность цикла — почему весь горячий путь неблокирующий

✓ разобрано · понимание подтверждено контрвопросами

Исходный вопрос

«Дальше про asyncio — это как раз про ту асинхронность, что обсуждали в самом начале?»

Главная мысль: агент почти всё время ЖДЁТ, а не считает

На каждом витке процессор не занят вычислениями — он ждёт I/O. Два главных ожидания: ответ модели (сетевой запрос к API, секунды) и работа инструментов (чтение файлов, shell). Это ожидание — не «зависание»: процесс не сломан, он просто стоит, пока не придёт ответ, чтобы продолжить.

Где async реально даёт выигрыш — только на нескольких ожиданиях сразу

Важная честная граница для нашего случая (один пользователь, один диалог по очереди):

  • Ожидание ОДНОГО ответа модели async не ускоряет. Ждать всё равно надо, и заняться в это время нечем: пользователь заблокирован ровно на том же ответе, отвечать ему пока нечем.
  • Fan-out тулов — вот где выигрыш. Модель за один ход просит несколько read-only инструментов; они по данным независимы (тема 1). Пускаем их через asyncio.gather — несколько ожиданий I/O накладываются друг на друга вместо суммирования: пока ждём первый файл, уже читаем второй и третий.

Почему тогда весь цикл на async, а не «местами»

Async «заразен». Событийный цикл один и кооперативный: корутины не вырывают управление силой, а сами отдают его на await. Пока одна честно ждёт через await — цикл крутит остальные. Но один блокирующий вызов (обычный time.sleep, синхронный requests.get, блокирующее чтение) управление НЕ отдаёт → цикл замирает целиком → «три файла параллельно» снова превращаются в три по очереди.

Отсюда правило: раз цикл асинхронный — весь горячий путь (запрос к модели, исполнение тулов) обязан быть неблокирующим, через await. Синхронную по природе работу нельзя звать напрямую в цикле — её уводят в отдельный поток (asyncio.to_thread), чтобы корутина честно отдала управление на время I/O.

Развёрнутое пояснение: асинхронность — не то же самое, что параллелизм на нескольких ядрах

Важно не спутать asyncio с многопроцессностью. Событийный цикл всё это время крутится на одном потоке (одном ядре процессора) — выигрыш идёт не от того, что что-то реально считается одновременно, а от того, что во время ожидания I/O процессор не простаивает впустую, а переключается на другую готовую к работе задачу. Если бы у агента была тяжёлая вычислительная работа (не I/O, а, скажем, разбор огромного AST на чистом Python), async ей бы никак не помог — она бы всё равно заняла процессор целиком и заблокировала событийный цикл не хуже синхронного вызова. Именно поэтому в конспекте выигрыш описан узко: async ускоряет ожидание (сеть, диск), а не вычисления. Для CPU-тяжёлой работы нужен был бы уже настоящий параллелизм — отдельные процессы, а не корутины.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: Что агент делает почти всё время — считает или ждёт? Чего именно ждёт?

О: Ждёт. Главные ожидания — ответ модели (API, секунды) и I/O инструментов. Процессор при этом простаивает, это не зависание.

В: При одном пользователе и одном диалоге — что даёт async?

О: На ожидании одного ответа модели — ничего (заняться нечем). Выигрыш только на fan-out: несколько read-only тулов разом через gather, их ожидания накладываются вместо суммирования.

В: Зачем весь цикл на async, если выигрыш только на параллельных тулах?

О: Async заразен: один блокирующий вызов в горячем пути замораживает весь единый event loop и убивает параллельность. Поэтому синхронную работу заворачивают в поток (to_thread), а сетевые вызовы делают await-ными.

19. Вставка прерываний в реальном времени — steering queue h2A

✓ разобрано · понимание подтверждено контрвопросами

Исходная идея

Ctrl+C посреди длинной задачи не роняет агента и не выбрасывает уже сделанную работу. Агент читает прерывание, аккуратно суммирует прогресс («что сделал / над чем работал / что осталось») и ждёт новой инструкции. Длинная задача перестаёт быть необратимым обязательством — её можно перенаправить на середине, добавить контекст, которого не было при старте, или попросить остановиться и отчитаться.

steering queue h2A — асинхронный канал сбоку от агентного цикла, который принимает сообщения из любого источника в любой момент. Цикл вычитывает из него в своих безопасных точках.

Почему канал — именно очередь, а не флаг и не прямой append

Ctrl+C во время работы агента ловится в input_loop — это отдельная асинхронная задача рядом с agent_task = asyncio.create_task(agent_loop(...)). То есть производитель сообщения (обработчик Ctrl+C) и потребитель (agent_loop) — разные места, работающие как бы одновременно.

  • Гонка двух писателей. В середине витка agent_loop активно трогает messages: дописывает ответ модели, собирает tool_result, снова дописывает. Если обработчик Ctrl+C сделает messages.append(...) напрямую — user-сообщение может воткнуться между tool_use и его tool_result. Для API это битая история → ошибка. Порядок сообщений — контракт, посторонний писатель его ломает.
  • Очередь развязывает producer и consumer. Ctrl+C только кладёт сообщение в interrupt_queue и messages не трогает. Перекладывает из очереди в messages только сам agent_loop — в своих безопасных точках (get_nowait()). Итог: у messages остаётся ровно один писатель — никаких гонок.
  • Канал несёт содержимое, а не «да/нет». Флаг interrupted = True сказал бы «стоп», но не что сказать агенту и не развязал бы момент записи. Очередь несёт текст прерывания, и источников может быть много (Ctrl+C, таймер, другой сокет) — все пишут в один asyncio.Queue, цикл разгребает единообразно.

Это ровно тема 1 (порядок и гонки на общем состоянии), только гонка теперь — между циклом и внешним прерывателем, и лечится тем же: у общего состояния остаётся один писатель.

Две точки проверки очереди

agent_loop вычитывает очередь в двух местах:

  1. Перед вызовом модели (начало витка).
  2. После ответа модели, но до запуска инструментов. Без этой проверки прерывание, пришедшее пока модель стримила ответ, не поймалось бы — и инструменты (возможно, тяжёлые или пишущие) уже запустились бы до следующего витка. Вторая проверка даёт остановиться перед исполнением тулов.

Остановка — на чистой границе, не мгновенно. Прерывание не обрывает работу посреди полузаписанного файла: агент дописывает текущий шаг и останавливается на безопасной границе (в примере статьи s03 был прочитан, но ещё не записан — агент честно сообщил это как «в процессе»). Так задумано: корректно завершить шаг, затем ждать новой инструкции.

Ценность модели взаимодействия

Длинные задачи остаются управляемыми, а не превращаются в «запустил и жди двадцать минут — вышло или нет». Можно перенаправить на середине (в примере: «пропусти s03, перейди к s04–s06»), не потеряв уже выполненное.

Развёрнутое пояснение: чем прерывание отличается от обычного нового сообщения пользователя

Формально прерывание и обычная реплика пользователя доставляются одним и тем же способом — через очередь, вычитываемую в безопасных точках. Разница не в канале, а в том, что происходит с уже запланированной работой. Обычное сообщение просто добавляется в конец контекста и ждёт своей очереди — текущие незавершённые намерения модели (например, оставшиеся пункты плана) никуда не деваются, модель может к ним вернуться. Прерывание же по смыслу (не по механике) — сигнал «остановись и переоцени», и системный промпт побуждает модель на прерывание явно суммировать прогресс и спросить, что делать дальше, а не молча продолжать план как ни в чём не бывало. То есть канал доставки один, но интерпретация сообщения как «прерывание» — это соглашение, зашитое в промпт, а не отдельная структурная сущность в очереди.

Проверка понимания (контрвопросы, на которые я ответил верно)

В: Почему канал прерываний — asyncio.Queue, а не флаг или прямой messages.append из обработчика Ctrl+C?

О: Producer (Ctrl+C в input_loop) и consumer (agent_loop) — разные задачи. Прямой append из producer создаёт гонку на messages и может воткнуть сообщение между tool_use и tool_result → битый контракт для API. Очередь оставляет единственным писателем сам цикл и несёт текст, а не просто «стоп».

В: Зачем проверять очередь дважды, а не только в начале витка?

О: Прерывание, пришедшее во время стрима ответа модели, иначе не поймалось бы — инструменты успели бы выполниться до следующей проверки. Вторая точка (после ответа, до тулов) даёт остановиться перед запуском возможно тяжёлых/пишущих инструментов.

В: Почему прерывание не обрывает работу мгновенно, посреди записи файла?

О: Так задумано — агент дописывает текущий шаг и останавливается на чистой границе, чтобы не оставить полузаписанный файл и точно суммировать сделанное.

20. Prompt caching и оптимизация KV-кэша

✓ разобрано · зафиксировано по просьбе

Исходная идея

Агент за одну длинную сессию делает сотни вызовов модели, и почти каждый вызов тащит один и тот же огромный префикс: системную инструкцию и определения инструментов. Разобранные трассы Claude Code показывают ~92% повторного использования префикса во внутренних вызовах — это не случайность, а следствие того, как собран промпт. Prompt caching обслуживает такой стабильный префикс примерно за 10% обычной стоимости токенов, поэтому экономия на длинной сессии огромна.

Правило раскладки промпта: стабильное содержимое — первым, изменяемое — последним. KV-кэш работает по префиксу: он переиспользуется ровно до первого различия. Поэтому всё, что не меняется между ходами (системная инструкция, определения инструментов), кладут в начало, а то, что растёт каждый ход (история сообщений, результаты тулов), — в хвост.

Почему кэшируется именно префикс

KV-кэш модели — это уже посчитанные ключи/значения внимания для токенов префикса. Если следующий запрос начинается с тех же токенов в том же порядке, эту часть внимания не надо считать заново — она берётся из кэша. Но как только встречается первый отличающийся токен, дальше всё считается с нуля. Отсюда прямое следствие: любая перестановка или вставка в начало промпта рушит кэш для всего хвоста. Стабильное — вперёд, летучее — назад.

Маркер cache_control

Провайдеру нужно явно сказать, докуда строить кэш-запись. Это делает маркер cache_control: {"type": "ephemeral"}, повешенный на последний блок стабильной части (последний блок системной инструкции и последнее определение инструмента). Смысл: «на первом вызове построй запись KV-кэша до этой точки; на последующих вызовах с тем же префиксом — обслуживай её из кэша». Всё, что до маркера, кэшируется; всё, что после, — считается каждый раз.

Экономика: MISS и HIT

  • Первый вызов — MISS. Кэша ещё нет, префикс оплачивается полностью и записывается в кэш (в трассе примера — 1 847 токенов «written»).
  • Последующие вызовы — HIT. Тот же префикс обслуживается из кэша за ~10% стоимости; в примере каждый HIT «экономит ~1 662 токена» (90% от 1 847).
  • Накопление. За 6 вызовов (5 попаданий) в примере сэкономлено ~8 310 токенов. В полной сессии с сотнями вызовов тот же механизм и даёт наблюдаемые в трассах 92% переиспользования префикса.

Отдельный трекер использования (счётчик created / read / uncached по полям usage каждого ответа) делает работу кэша видимой: на каждом ходу понятно, был MISS или HIT и сколько токенов сэкономлено, а в конце — суммарная экономия. Наблюдаемость здесь та же идея, что и с хуками (тема «шина событий»): метрику выносят сбоку, не вплетая в горячий путь.

Связь с остальным

Кэш накладывает дисциплину на структуру промпта, и она согласуется с уже пройденным: постепенное раскрытие инструментов и правка системной инструкции опасны не только логически, но и по деньгам — меняя начало промпта, ты инвалидируешь кэш всего хвоста. Стабильность префикса становится не эстетикой, а прямой экономией.

Развёрнутое пояснение: почему кэш не помогает при ветвлении разговора

Стоит явно связать эту тему с темой 17 (ветвление сессий). Казалось бы, ветка — это «тот же префикс, что и раньше», значит кэш должен работать безотказно. Но кэш чувствителен не только к тексту, а и к порядку запросов: если между сохранением ветки и возвратом к ней прошло много других вызовов (в том числе к другим сессиям), запись в кэше провайдера могла устареть и вытесниться — у KV-кэша обычно есть TTL (время жизни) порядка нескольких минут. Возврат к старой ветке через /resume после долгого перерыва почти наверняка означает MISS на первом вызове, даже если сам префикс токен-в-токен идентичен тому, что кэшировался раньше. Экономия prompt caching устойчива внутри активной, непрерывно используемой сессии — и не гарантирована для истории, к которой возвращаются спустя часы или дни.

21. Интеграция с MCP-серверами

✓ разобрано · зафиксировано по просьбе

Что такое MCP для агента

MCP (Model Context Protocol) — стандартный способ подключать внешние серверы инструментов. Сервер файловой системы даёт файловые инструменты, git-сервер — операции git, сервер БД — запросы. Ключевая идея: инструменты любого MCP-совместимого сервера становятся полноправными участниками реестра инструментов агента, наравне со встроенными. Модель вызывает их точно так же и не знает, локальная это Python-функция или удалённый серверный процесс.

Как это заводится на старте

  • Runtime читает конфиг серверов из config/mcp_config.yaml.
  • При запуске подключается к каждому серверу через официальный Python SDK MCP (транспорт stdio: поднимается процесс, устанавливается ClientSession, вызывается initialize()).
  • Вызывает list_tools(), чтобы обнаружить предоставляемые сервером инструменты.
  • Регистрирует каждый под префиксным именем mcp__<server>__<tool> в той же карте диспетчеризации, где лежат встроенные, и отдаёт модели их описания и input_schema.

Префикс — это только деталь маршрутизации. Имя mcp__git__git_log нужно диспетчеру, чтобы понять, к какому серверу направить вызов. Для модели же вызов MCP-инструмента ничем не отличается от вызова bash или grep: та же структура tool call, та же вставка результата, тот же цикл.

Диспетчеризация: одна проверка префикса

Маршрутизатор обрабатывает MCP-инструменты так же, как встроенные, — вся разница в одной проверке: если имя начинается с mcp__, вызов уходит в call_mcp_tool (тот находит нужную сессию по имени сервера и зовёт session.call_tool); иначе берётся локальный обработчик из карты диспетчеризации. Результат в обоих случаях возвращается в цикл одинаково — как строка. Сам агентный цикл не меняется.

Что видит модель

В примере s21 при подключении filesystem- и git-серверов реестр просто вырос: 14 встроенных + 14 MCP = 28 инструментов. Модель вызвала mcp__git__git_log, mcp__git__git_diff и mcp__filesystem__read_file ровно так же, как звала бы встроенные, и даже смешала MCP-вызов с bash в параллельном запуске. Никакой разницы в маршрутизации, видимой модели, нет.

Связь с остальным

MCP — это чистое расширение реестра инструментов без вторжения в горячий путь: сервера прозрачно добавляют инструменты, не трогая ни цикл, ни схему взаимодействия модели. Тот же принцип, что и раньше: агентный цикл остаётся простым, а возможности наращиваются по краям (инструменты, хуки, кэш), а не переписыванием ядра.

Развёрнутое пояснение: цена, которую MCP добавляет незаметно

Прозрачность MCP для модели не значит, что подключение сервера бесплатно для системы в целом. Во-первых, каждый подключённый сервер добавляет свои схемы инструментов в промпт — а значит, сдвигает границу «мало тулов / сотни тулов» из темы про постепенное раскрытие: подключив пять MCP-серверов по десять инструментов каждый, легко незаметно перейти в режим, где полные схемы уже стоят заметных денег на каждом вызове. Во-вторых, MCP-сервер — это отдельный процесс, который тоже может зависнуть или упасть; поскольку вызов идёт по тому же принципу, что и обычный инструмент, зависший MCP-сервер блокирует виток цикла ровно как зависший обычный tool call — асинхронность из темы 18 защищает от блокировки другими ожиданиями, но не от того, что сам вызов не возвращается вовсе. Расширяемость реестра — не бесплатная суперспособность, а тот же компромисс «больше возможностей — больше поверхность отказа», что и у любого добавленного инструмента.

22. Корпоративные улучшения: промышленные почтовые ящики и жизненный цикл worktree

✓ разобрано · зафиксировано по просьбе

Фаза посвящена одному: закрыть разрыв между рабочим прототипом и системой, которую можно запускать в промышленной среде. Учебные реализации заменяются промышленными альтернативами — при неизменной логике агентов. Два крупных куска: почтовые ящики (транспорт координации) и жизненный цикл worktree (изоляция параллельной правки).

Часть 1. Промышленные почтовые ящики: Redis Pub/Sub вместо JSONL

JSONL-почтовые ящики из фазы 4 хороши как учебный механизм, но имеют три фундаментальных промышленных изъяна:

  • Опрос (polling). Товарищ проверяет входящий файл по таймеру → искусственная задержка между отправкой и получением.
  • Файловые блокировки. Два агента, пишущие в один файл одновременно, повреждают JSONL — нужен lock на конкурентный доступ.
  • Одномашинность. Файлы ящиков живут в одной ФС → распределённое развёртывание невозможно.

Внутренняя координация агентов Claude Code опирается на обмен сообщениями, который мгновенный, без блокировок и работает через границы процессов. Redis Pub/Sub даёт ровно эти три свойства: агент публикует в канал, любой подписчик получает сообщение за миллисекунды — без опроса, без файловых локов, без зависимости от файловой системы.

Ключ — общий интерфейс, скрывающий транспорт

Оба бэкенда прячутся за одним абстрактным интерфейсом MailboxBackend с тремя методами: send(), receive(), close(). Поэтому логика товарищей и ведущего агента из фазы 4 остаётся полностью неизменной — они просто зовут send() / receive() и не знают, как это реализовано.

  • RedisMailbox — промышленный бэкенд. Канал = agent:<name>:inbox; send = redis.publish(channel, payload); receive подписывается на канал и тянет сообщения через pubsub.get_message до дедлайна по таймауту. Мгновенная доставка, кросс-машинность.
  • QueueMailbox — резервный бэкенд для локальной разработки, когда Redis недоступен: внутрипроцессная asyncio.Queue. Тот же интерфейс, но без нескольких машин и без мгновенной доставки.
  • make_mailbox() — фабрика: пробует Redis (redis.ping()), при неудаче откатывается на Queue. Так система работает и с Redis, и без него, не меняя код агентов.

Цикл товарища (teammate_loop) идентичен фазе 4: получил сообщение из ящика → прогнал через модель и инструменты (полный агентный цикл) → отправил результат обратно отправителю тем же mailbox.send(). Интерфейс ящика скрывает все детали транспорта; можно запускать нескольких товарищей параллельно, каждого на своём Redis-канале.

Что даёт замена (трасса s22)

Ведущий разослал задачи двум специалистам (explorer ищет пробелы в тестовом покрытии core.py, writer пишет тесты, чтобы их закрыть) — параллельно, каждый на своём канале. Итог: explorer нашёл 3 непокрытых ветки (88%), writer довёл покрытие до 100%.

Разница в задержке видна прямо в выводе: Redis доставляет за <10 мс, тогда как JSONL ограничен ~500-мс интервалом опроса. В сессии с десятками межагентных сообщений это складывается в заметную экономию wall-time. Важнее скорости: Redis-бэкенд работает одинаково, запущены агенты на одной машине или на разных машинах кластера — JSONL этого не умеет в принципе.

Часть 2. Расширенное управление жизненным циклом worktree

Базовый worktree из фазы 4 создаёт и удаляет ветку, но молча ломается на каждом крайнем случае реальной кодовой базы:

  • Репозиторий с незакоммиченными изменениями ведёт себя иначе при создании worktree (ветвление идёт от HEAD, а не от рабочего дерева).
  • Ветка от предыдущего упавшего запуска уже существует → git worktree add падает.
  • Репозиторий в состоянии detached HEAD не может создавать новые ветки.
  • Два агента, правящие один файл в разных ветках, создают конфликт слияния, который ни один агент сам не поймает.

Сам Claude Code избегает большинства этих проблем, используя снимки файлов (снапшоты, тема 14) вместо worktree. Но worktree даёт более сильную изоляцию: агент буквально не может тронуть файлы другого агента. Поэтому для крупных параллельных задач это правильный выбор — а промышленный менеджер обязан системно обработать каждый крайний случай до начала выполнения.

Из чего состоит промышленный менеджер worktree

  • check_git_state() — полная предполётная проверка перед любыми операциями: текущая ветка, «грязный» ли рабочий каталог (status --porcelain), detached ли HEAD, сколько уже worktree.
  • prune_stale_worktrees() — удаляет worktree, чьи каталоги уже не существуют (остатки от прошлых падений), затем git worktree prune.
  • create_worktree_safe() — безопасная обёртка над созданием: отказывается при detached HEAD (с понятной ошибкой), предупреждает о незакоммиченных изменениях, чистит путь от прошлого запуска, разрешает конфликт имени ветки (удаляет старую или берёт альтернативное имя с uuid-суффиксом), и только потом worktree add -b. Возвращает (path, branch) или бросает RuntimeError с ясным сообщением.
  • detect_conflicts() — после выполнения проверяет, изменяли ли параллельные задачи пересекающиеся файлы (git diff --name-only по веткам, пересечение множеств). Если да — можно быстро остановиться или предупредить.

Почему «понятная ошибка» критична для UX: если модель попытается создать worktree, а он молча не создастся из-за крайнего случая, агент запутается и, скорее всего, провалит задачу без объяснения причины. Явная RuntimeError с текстом («сначала git checkout main») даёт модели шанс исправиться.

Гарантированная очистка через try/finally

run_task_in_worktree_safe() оборачивает всё выполнение задачи в try/finally: в finally worktree и ветка удаляются всегда — даже при исключении. Внутри try крутится обычный агентный цикл, но bash-инструмент выполняется с chdir в каталог worktree, так что правки идут в изолированную ветку.

Трасса s23: две задачи (валидация run_bash и счётчик вызовов) выполнились полностью параллельно в изолированных каталогах, каждая независимо прогнала свои тесты (44 passed) — потому что писали в разные worktree. После завершения detect_conflicts корректно выявил пересечение (обе тронули core.py). Ветки при этом сохранены, а не удалены, чтобы слияние выполнить осознанно вручную.

Общий смысл фазы

Обе замены следуют одному принципу, уже знакомому по прошлым темам: ядро (агентный цикл, логика товарищей) не переписывается — меняется только реализация «по краям». Redis прячется за тем же MailboxBackend, что и Queue; промышленный worktree — это тот же контракт «создать изоляцию → выполнить → гарантированно убрать», просто с обработкой каждого крайнего случая и режима отказа. Промышленная зрелость = «каждый edge case обработан, каждый режим отказа учтён, очистка гарантирована независимо от происходящего».

Развёрнутое пояснение: почему Redis добавляет новую зависимость, которую тоже нужно эксплуатировать

Замена JSONL на Redis решает три изъяна учебного варианта (опрос, файловые локи, одномашинность), но платит за это тем, что появляется внешняя система, от которой теперь зависит вся координация. Если Redis недоступен (упал, сеть отвалилась), make_mailbox() откатывается на QueueMailbox — но это откат к внутрипроцессной очереди, а значит и к одномашинности, ради избавления от которой Redis и подключали. То есть промышленный вариант не убирает компромисс «простота против масштабируемости» — он лишь делает его настраиваемым и явным (через фабрику), а не зашитым намертво в код. Это общее свойство любой промышленной замены учебного механизма: внешняя зависимость даёт мощь ценой того, что теперь нужно следить за её здоровьем отдельно от здоровья самих агентов.

23. Как улучшить дальше — дорожная карта

✓ зафиксировано по просьбе · направления развития, не реализация

На этом обвязка Claude Code собрана целиком: от минимального агентного цикла до промышленной многоагентной системы — потоковая выдача, параллельное выполнение, кэш промпта, Redis-почтовые ящики, управление разрешениями, сохранение сессий и официальная MCP-среда. Архитектура чистая, без повторов, покрыта тестами. Пять направлений, куда её можно развивать дальше:

1. Параллельный запуск подагентов

Сейчас подагенты запускаются последовательно. Рефакторинг spawn_subagent на asyncio.gather дал бы ведущему запускать, скажем, три исследовательских подагента одновременно (именно так делает сам Claude Code) — время исследования сокращается пропорционально числу параллельных агентов.

2. Векторное хранилище памяти

Долговременная память — плоский markdown-файл, который целиком вставляется в каждую сессию. Замена на лёгкое векторное хранилище (например ChromaDB) позволила бы доставать семантически релевантные воспоминания, а не всю сводку сразу — контекст остаётся сфокусированным по мере роста проекта.

3. Точный учёт токенов

Трекер статистики кэша считает токены по сессии, но не раскладывает стоимость по задачам или типам инструментов. Журнал затрат по операциям показал бы, какие вызовы дороже всего, и где оптимизировать.

4. Шина событий на webhooks

Текущая шина запускает хуки только внутри процесса. Если добавить отправку событий во внешний HTTP-endpoint, появляется интеграция со Slack, Datadog, PagerDuty или любым мониторингом — без изменения агентного цикла (тот же принцип «меняем края, ядро не трогаем»).

5. Фреймворк оценки

Тесты проверяют, что обвязка работает корректно, но не измеряют, насколько хорошо агент решает реальные задачи. Слой оценки LLM-as-a-judge — оценка ответов по точности, эффективности использования инструментов и следованию плану — превратил бы репозиторий из просто рабочей системы в систему, пригодную для бенчмарков.

Общая нить всех пяти пунктов та же, что и у фазы 6: улучшения ложатся на края (транспорт памяти, шина, слой оценки, планировщик подагентов), а ядро — агентный цикл — остаётся нетронутым.

Развёрнутое пояснение: почему все пять пунктов совместимы друг с другом

Стоит заметить, что пять направлений не конкурируют за одно и то же место в системе — они трогают пять разных, невзаимодействующих краёв: планировщик подагентов (вход в один инструмент), хранилище памяти (то, что грузится при старте сессии), учёт токенов (наблюдатель сбоку от вызовов), шина событий (уже готовая точка расширения из темы 16) и слой оценки (отдельный процесс поверх готовых трасс). Поэтому их можно внедрять в любом порядке и любой комбинации, не переделывая уже сделанное, — ровно то же свойство «независимости по краям», что уже встречалось у хуков и MCP: расширения не знают друг о друге и не обязаны согласовываться между собой, чтобы сосуществовать.
